# Re:Fail 포트폴리오 개선 기록

## 1. 문서 목적

이 문서는 Re:Fail을 기획하고 구현하면서 발견한 문제와 개선 과정을 포트폴리오, README, 발표 자료, 면접 답변에 재사용하기 위해 작성한다.

각 개선은 다음 순서로 기록한다.

> 문제 → 원인과 제약 → 선택 → 구현 → 검증 → 결과와 한계

기능 개수보다 어떤 문제를 발견했고, 왜 그 방법을 선택했으며, 어떻게 검증했는지를 설명하는 것을 목표로 한다.

## 2. 프로젝트 한눈에 보기

### 서비스 소개

**Re:Fail - 실패를 공유하고, 다음을 준비하다**

성공과 행복이 주로 노출되는 SNS 환경에서 사용자가 크고 작은 실패를 안전하게 기록하고, 후속 기록을 통해 다시 시도하거나 극복하는 과정을 남기는 서비스다. 불행을 전시하는 것이 아니라 실패를 돌아보고 같은 실수를 반복하지 않도록 돕는 것이 목적이다.

### 핵심 사용자 흐름

1. 사용자가 실패 경험을 마크다운으로 기록한다.
2. 익명 또는 닉네임 공개 방식을 선택한다.
3. 다른 사용자는 조언 대신 정해진 공감 반응을 남긴다.
4. 작성자는 `다시 시도 중`, `잠시 멈춤`, `조금씩 나아지는 중`, `극복함`과 같은 후속 기록을 추가한다.
5. 하나의 실패가 끝난 사건이 아니라 변화와 성장의 흐름으로 남는다.

## 4. 핵심 개선 사례

### 4.1 익명성과 연결성을 함께 유지한 공개 정책

**문제**

실패는 성공보다 공개 부담이 크다. 완전 실명제는 작성 장벽이 높고, 완전 익명만 허용하면 같은 사용자의 변화와 후속 기록을 연결하기 어렵다.

**선택**

게시글마다 익명 또는 닉네임 공개를 선택할 수 있게 했다. 공개 화면에서는 선택한 신원만 보여주지만, 서버는 JWT 사용자 ID로 실제 작성자를 관리한다.

**구현**

- 게시글에 공개 유형을 저장한다.
- 응답 DTO에서 익명 글은 작성자 정보를 노출하지 않는다.
- 수정·삭제·후속 기록 작성은 공개 이름이 아니라 인증 사용자 ID로 검증한다.
- 익명 글도 `내 기록 보관함`에서는 작성자 본인이 조회하고 관리할 수 있다.

**결과**

감정적 안전을 위한 익명성과 성장 기록을 이어가기 위한 계정 연속성을 동시에 확보했다.

### 4.2 댓글 대신 제한된 공감 반응을 선택

**문제**

실패 경험에 달리는 조롱, 과도한 충고, 비난은 일반 게시판보다 더 큰 상처가 될 수 있다. 댓글을 추가하면 실시간 운영과 moderation 범위도 크게 늘어난다.

**선택**

MVP에서는 댓글을 보류하고 서비스가 정의한 공감 반응만 제공했다.

**구현**

- 사용자당 게시글 하나의 공감만 유지한다.
- 반응별 개수를 집계한다.
- 자신의 글에는 공감할 수 없게 서버에서 제한한다.
- 작성자가 원하는 반응을 우선 노출할 수 있게 했다.
- 부적절한 게시글은 별도의 신고 흐름으로 처리한다.

**결과와 한계**

상호작용은 유지하면서 공격 표면과 운영 부담을 줄였다. 실제 사용자 검증 후 안전 정책과 운영 역량이 준비되면 제한형 댓글을 다시 검토할 수 있다.

### 4.3 실패에서 성장으로 이어지는 후속 기록

**문제**

실패 글만 모으면 서비스가 부정적인 경험을 전시하는 공간으로 보일 수 있다.

**선택**

성공 글을 독립 게시물로 만들지 않고 최초 실패 글의 후속 기록으로 연결했다.

**구현**

- 후속 기록 작성·조회·수정·삭제 API를 구현했다.
- 재시도, 중단, 개선, 극복 상태를 저장한다.
- 목록에서 후속 기록 존재 여부를 표시한다.
- 작성 단계에 선택 항목인 `다음에는 어떤 선택을 해볼 것인가`를 추가했다.
- 운영 지표에서 후속 기록 작성률과 상태 비율을 확인할 수 있게 했다.

**결과**

`실패 작성 → 다시 시도 → 변화 기록`이라는 Re:Fail만의 핵심 흐름이 완성됐다.

### 4.4 JWT 기반 CRUD 권한 검증

**문제**

프론트엔드에서 수정·삭제 버튼을 숨기는 것만으로는 다른 사용자가 API를 직접 호출하는 것을 막을 수 없다.

**선택**

모든 변경 API에서 JWT의 사용자 ID와 리소스 작성자 ID를 서버가 비교하도록 했다.

**구현**

- Spring Security 필터에서 Bearer 토큰을 검증한다.
- 컨트롤러에는 인증 사용자 정보를 인자로 전달한다.
- 게시글과 후속 기록의 수정·삭제 전에 소유권을 검증한다.
- 제한된 사용자는 작성과 상호작용을 차단한다.
- 관리자 API는 사용자 역할까지 추가로 검증한다.

**검증**

정상 작성자, 다른 사용자, 비로그인 사용자 요청을 통합 테스트로 구분해 검증했다.

### 4.5 신고와 관리자 moderation 흐름

**문제**

사용자 콘텐츠 서비스는 악성 게시물과 허위·중복 신고를 처리할 수 있어야 한다.

**선택**

신고 접수와 게시글 삭제를 바로 연결하지 않고 관리자가 판단한 후 숨김 처리하는 구조를 선택했다.

**구현**

- 사용자·대상 조합의 중복 신고를 데이터베이스 제약과 서비스 검증으로 방지했다.
- 관리자가 상태별 신고 목록과 상세 사유를 조회한다.
- 게시글을 소프트 숨김하고 필요하면 복구한다.
- 처리자, 처리 시점, 사유를 운영 이력으로 남긴다.
- 숨김 시 대기 중인 관련 신고를 함께 처리한다.

**결과**

사용자 신고, 관리자 판단, 콘텐츠 노출 제어, 감사 이력이 하나의 흐름으로 연결됐다.

### 4.6 Flyway 기반 MySQL 스키마 관리

**문제**

JPA의 자동 스키마 변경에 의존하면 개발·시연·운영 환경의 테이블 구조가 달라질 수 있고 변경 이력을 설명하기 어렵다.

**선택**

MySQL 8을 기준 데이터베이스로 사용하고 Flyway로 스키마 버전을 관리한다. Hibernate는 스키마를 수정하지 않고 검증만 한다.

**구현**

- `V1`: 초기 사용자, 게시글, 후속 기록, 공감, 신고, 운영 이력 테이블
- `V2`: 기본 카테고리 데이터
- `V3`: 다음 시도 계획 필드
- `V4`: 읽기 성능 복합 인덱스
- `ddl-auto=validate`로 엔티티와 실제 스키마 불일치를 시작 단계에서 탐지

**결과**

데이터베이스 변경이 코드와 함께 버전 관리되고, 새로운 환경에서도 같은 순서로 재현된다.

### 4.7 게시글 조회 성능 개선

**문제**

게시글 목록 DTO를 만드는 과정에서 지연 로딩된 작성자와 카테고리를 게시글마다 조회해 N+1 쿼리가 발생했다. 관리자 신고 목록에서도 같은 문제가 있었다.

**원인**

목록 쿼리는 `Post`만 조회했지만 응답 변환 단계에서 `Post.user`, `Post.category`에 접근했다. 후속 기록 존재 여부도 별도 조회가 필요했다.

**선택과 구현**

- `EntityGraph`로 게시글·작성자·카테고리를 함께 조회했다.
- 현재 페이지의 게시글 ID를 모아 후속 기록 존재 여부를 한 번에 조회했다.
- 신고 목록도 신고자 정보를 함께 조회했다.
- 카테고리 목록을 애플리케이션 메모리에 캐시하고 관리자 변경 시 무효화했다.
- 실제 조건에 맞춘 세 개의 MySQL 복합 인덱스를 추가했다.
- 운영 기본 SQL 디버그 로그를 끄고 batch fetch와 JDBC batch 설정을 적용했다.

**검증**

- 게시글 10개를 조회해도 준비된 SQL이 최대 3회인지 통합 테스트로 검증했다.
- Flyway `v4`와 인덱스가 실제 MySQL에 적용됐는지 확인했다.
- 워밍업 후 로컬에서 API별 5회 응답 시간을 측정했다.

| API | 평균 응답 시간 |
| --- | ---: |
| 게시글 목록 | 26.29ms |
| 카테고리 목록 | 12.38ms |
| 헬스 체크 | 13.08ms |

**한계와 다음 단계**

제목·본문 검색은 `%키워드%` 검색이라 일반 B-Tree 인덱스를 충분히 활용하지 못한다. 실제 데이터가 수만 건 이상 쌓여 병목이 확인되면 MySQL FULLTEXT를 도입하고, 데이터 규모와 검색 요구가 더 커질 때 별도 검색 엔진을 검토한다.

### 4.8 운영 전 P1 위험 개선

**문제**

코드 리뷰와 실행 API 검증에서 기능 테스트만으로 발견하지 못한 다섯 가지 운영 위험을 확인했다. 공감·신고 카운터의 동시성 정합성, 숨김 콘텐츠의 하위 기록 노출, 제한된 관리자 토큰, 검색어별 무제한 캐시, 기본 JWT 시크릿 문제였다.

**선택과 구현**

- 공감·신고 카운터는 엔티티의 읽기-수정-쓰기 대신 DB의 `count = count ± 1` 원자적 UPDATE를 사용했다.
- 벌크 UPDATE 후 영속성 컨텍스트를 clear해 같은 트랜잭션에서도 이전 카운터를 읽지 않게 했다.
- 공개 후속 기록 API는 부모 게시글이 숨김·삭제 상태인지 먼저 검증한다.
- 모든 관리자 API는 토큰 claim뿐 아니라 DB의 현재 역할과 `ACTIVE` 상태를 확인한다.
- 변경이 적은 전체 카테고리 목록만 캐시하고 사용자 검색 결과는 캐시하지 않는다.
- 운영 기본 JWT 시크릿을 제거하고 개발용 값은 `local`·`docker` 프로필에만 격리했다.
- JWT 서명·만료와 함께 issuer도 검증한다.

**검증**

- 전체 14개 백엔드 테스트를 캐시 없이 재실행했다.
- 실제 MySQL에서 사용자 5명이 동시에 공감·신고하도록 요청했다.
- 공감과 신고 모두 HTTP 요청 5건이 성공했고, DB 카운터 5와 실제 행 5가 일치했다.
- 숨김 게시글 후속 기록은 `404`, 제한 관리자 토큰의 조회·변경은 `403`임을 실행 API에서 확인했다.
- 검증용 사용자와 게시글은 확인 후 데이터베이스에서 제거했다.

**성과**

정상 흐름 중심의 기능 완성에서 한 단계 더 나아가 동시성, 권한 회수, 캐시 키 공간, secure default를 테스트와 실제 MySQL 결과로 설명할 수 있게 됐다.

### 4.9 실제 MySQL 자동 검증과 재현 가능한 배포

**문제**

H2 MySQL 호환 모드는 SQL 문법 일부를 확인할 수 있지만 InnoDB의 잠금과 교착 상태, Flyway의 실제 MySQL 적용을 보장하지 못한다. 또한 개발 환경에서만 실행되면 포트폴리오 검증자가 같은 구성을 재현하기 어렵다.

**선택과 구현**

- Testcontainers MySQL 8.4를 전체 Gradle 테스트에 포함했다.
- 공감·신고 각각 8개 동시 요청과 집계 값, 실제 행 개수를 함께 검증했다.
- 테스트 중 발견한 자식 행 INSERT와 부모 집계 UPDATE 사이의 InnoDB 교착 상태를 잠금 순서 통일로 해결했다.
- Flyway 마이그레이션 수와 사용자 운영 이력의 대상 유형을 실제 MySQL에서 검증했다.
- 백엔드와 프론트엔드를 다단계 Docker 이미지로 만들고 MySQL을 포함한 Compose 구성을 추가했다.
- 운영 프로필에서 Swagger를 기본 비활성화하고 시크릿은 환경 변수로만 주입한다.

**대안 비교**

- H2만 유지하면 테스트는 빠르지만 InnoDB 잠금과 MySQL 마이그레이션 호환성을 검증할 수 없다.
- 공용 테스트 DB를 사용하면 실제 MySQL을 쓸 수 있지만 테스트 간 데이터 격리와 CI 재현성이 낮아진다.
- Testcontainers는 Docker 시작 비용이 추가되지만 테스트마다 동일한 MySQL 버전과 독립된 스키마를 재현할 수 있어 선택했다.
- 로컬 Java·Node 프로세스만 실행하는 방식보다 Docker 이미지 빌드 비용은 늘지만, 실행 환경과 비루트 사용자·헬스 체크까지 함께 검증할 수 있어 Compose를 제공했다.

**관측성**

- 모든 요청에 `X-Request-ID`를 부여하고 MDC 로그에 처리 시간과 상태 코드를 남긴다.
- 공개 헬스 체크와 관리자 전용 Micrometer·Prometheus 엔드포인트를 분리한다.
- 목록 크기와 검색어 길이에 상한을 적용해 과도한 조회 비용과 캐시 키 증가를 방지한다.

**검증 결과**

- 전체 16개 테스트 스위트의 자동 테스트 27개가 실패·오류 없이 통과했다.
- 이 중 실제 MySQL 핵심 테스트 4개가 Flyway, 동시성, 운영 이력, 상태 정책, 목록 조회를 검증한다.
- MySQL, Spring Boot, Next.js 세 컨테이너가 모두 `healthy` 상태인지 확인했다.
- 운영 프로필에서 헬스 체크는 `200`, OpenAPI 경로는 `404`인지 확인했다.
- 프론트엔드 SSR이 Compose 내부 주소로 백엔드 데이터를 조회하는지 확인했다.
- 실제 MySQL에서 동시 공감·신고 집계와 행 개수가 각각 8로 일치했다.

**개선 전후와 남은 한계**

- 개선 전에는 동시 요청 중 InnoDB 교착 상태가 발생했지만, 부모 집계 UPDATE와 자식 행 변경의 잠금 순서를 통일한 뒤 동일 테스트가 통과했다.
- 수동 MySQL 검증에서 끝나던 동시성 확인을 Gradle과 GitHub Actions에서 반복 가능한 자동 테스트로 전환했다.
- Testcontainers는 Docker가 필요하고 전체 테스트 시간이 증가한다. 검색 데이터가 커질 때의 FULLTEXT 성능과 장시간 부하 테스트는 별도 환경에서 검증해야 한다.

## 5. 기술 선택 근거

### Spring Boot 단일 서버

포트폴리오 MVP에서 트래픽보다 중요한 것은 핵심 흐름을 완성하고 운영 가능한 구조를 보여주는 것이다. 인증, 게시글, 공감, 신고를 마이크로서비스로 분리하면 배포·관측·트랜잭션 복잡도가 먼저 증가한다. 따라서 도메인 패키지로 경계를 나눈 모듈형 단일 서버를 선택했다.

### 로컬 메모리 캐시

카테고리는 조회가 많고 변경이 적으며 현재 서버가 한 대다. Redis를 추가하면 비용과 장애 지점이 늘어나므로 로컬 캐시를 사용했다. 다중 서버로 확장하는 시점에 공유 캐시가 필요하다는 전환 조건도 문서화했다.

### 이미지·DM·팔로우 제외

세 기능은 실패 기록과 후속 성장이라는 핵심 경험에 필수적이지 않다. 이미지 저장 비용, 개인 간 괴롭힘 대응, 인기 경쟁 유도 같은 운영 문제를 먼저 만든다. 기능 부족이 아니라 MVP 목적을 지키기 위한 의도적인 범위 관리다.

### 댓글 보류

댓글은 상호작용을 늘리지만 감정적으로 취약한 콘텐츠에서 공격 위험이 크다. 안전한 운영 규칙과 moderation 비용을 검증하기 전까지 제한된 공감 반응만 제공한다.

## 7. 포트폴리오에 바로 쓰는 문장

### 한 줄 소개

> 성공 중심 SNS의 비교 피로를 문제로 정의하고, 실패 기록과 후속 성장 과정을 안전하게 공유하는 서비스 Re:Fail을 기획·구현했습니다.

### 백엔드 요약

> Java 21과 Spring Boot 3 기반의 단일 서버를 설계하고 JWT 인증, 게시글·후속 기록 CRUD, 공감, 신고, 관리자 moderation을 구현했습니다. MySQL 스키마는 Flyway로 버전 관리하고 통합 테스트로 권한과 주요 사용자 흐름을 검증했습니다.

### 성능 개선 요약

> 게시글과 신고 목록에서 JPA 지연 로딩으로 발생한 N+1 문제를 발견해 EntityGraph와 일괄 조회로 개선했습니다. 게시글 수와 무관하게 목록 조회 SQL을 최대 3회로 제한하는 회귀 테스트를 추가하고, 실제 조회 조건에 맞춘 MySQL 복합 인덱스와 카테고리 캐시를 적용했습니다.

### 정책 설계 요약

> 실패 공유의 심리적 장벽과 악성 상호작용 위험을 고려해 익명·닉네임 선택형 정책을 설계했습니다. 댓글·DM·팔로우를 의도적으로 제외하고 제한된 공감 반응과 신고·숨김 흐름을 우선 구현했습니다.

### 프로젝트에서 강조할 역량

- 요구사항을 기능 목록이 아니라 사용자 문제와 운영 위험으로 정의한 경험
- 인증과 서버 권한 검증을 포함한 실제 CRUD 설계 경험
- 신고, 숨김, 복구, 감사 이력을 연결한 운영 기능 구현 경험
- Flyway를 이용한 재현 가능한 데이터베이스 변경 관리 경험
- N+1 탐지, 쿼리 수 회귀 테스트, 인덱스와 캐시 적용 경험
- 기능을 무작정 추가하지 않고 MVP 범위를 근거 있게 제한한 경험

## 8. 면접 질문 대비

### 왜 완전 익명 서비스가 아닌가요?

완전 익명은 작성 장벽을 낮추지만 실패 이후의 변화와 후속 기록을 같은 사용자 흐름으로 연결하기 어렵습니다. 그래서 외부 노출은 익명·닉네임 중 선택하게 하고, 내부적으로는 인증 계정에 연결해 본인 관리와 권한 검증이 가능하게 했습니다.

### 왜 댓글을 구현하지 않았나요?

개발 난이도보다 서비스 안전 문제를 우선했습니다. 실패 경험에는 조롱과 과도한 충고가 더 큰 피해가 될 수 있고 댓글은 실시간 moderation 범위를 크게 늘립니다. MVP에서는 제한된 공감과 신고 흐름으로 핵심 가설을 먼저 검증하도록 범위를 정했습니다.

### 왜 마이크로서비스를 사용하지 않았나요?

현재 규모에서는 서비스 분리로 얻는 독립 확장성보다 배포, 네트워크, 분산 트랜잭션 복잡도가 더 큽니다. 하나의 Spring Boot 서버 안에서 도메인 패키지 경계를 명확히 나누고, 실제 확장 필요가 생길 때 분리할 수 있도록 설계했습니다.

### N+1 문제를 어떻게 검증했나요?

응답 DTO 변환 시 지연 로딩 연관 관계에 접근하는 경로를 확인하고 Hibernate Statistics로 실행 SQL 수를 측정했습니다. 게시글 10건을 조회해도 SQL이 최대 3회를 넘지 않는 통합 테스트를 추가해 이후 변경에서도 성능이 퇴행하지 않도록 했습니다.

### 왜 모든 컬럼에 인덱스를 추가하지 않았나요?

인덱스는 조회를 빠르게 하지만 쓰기와 저장 공간 비용을 증가시킵니다. 실제 필터와 정렬 조건인 사용자 기록, 실패 크기, 후속 상태에만 복합 인덱스를 추가했습니다. `%키워드%` 검색은 일반 인덱스로 해결되지 않으므로 데이터 증가와 측정 결과를 기준으로 FULLTEXT 도입을 판단하도록 남겼습니다.
