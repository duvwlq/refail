# 프로젝트 하네스 문서

## 1. 문서 목적

이 문서는 `Fail` 프로젝트를 구현하거나 설명할 때 빠르게 참고하는 기준 문서다.

상세 기획은 [PLAN.md](D:/Codex_Project/Fail/PLAN.md), MVP 범위는 [MVP_SPEC.md](D:/Codex_Project/Fail/MVP_SPEC.md)를 기준으로 한다.

이 문서는 다음을 빠르게 맞추기 위해 사용한다.
- 이 프로젝트가 무엇을 해결하는지
- 무엇을 반드시 지켜야 하는지
- MVP에서 어디까지 구현하는지
- 어떤 판단을 하면 안 되는지

## 2. 프로젝트 한 줄 요약

실패를 공유하고, 공감받고, 나중에 변화나 성공까지 이어서 기록할 수 있게 만드는 공감형 커뮤니티 서비스

## 3. 이 프로젝트의 본질

이 프로젝트는 단순 게시판이 아니다.

핵심은 `실패를 덜 무섭게 느끼게 만드는 제품 설계`다.

따라서 구현 시에도 다음 우선순위를 지켜야 한다.
1. 감정적 안전
2. 글쓰기 진입장벽 완화
3. 실패 이후의 연속성
4. 과하지 않은 기능 범위
5. 포트폴리오 설명 가능성

## 4. 반드시 유지할 핵심 원칙

### 서비스 원칙
- 실패는 조롱 대상이 아니다
- 큰 실패와 작은 실패를 같은 공간에서 다룬다
- 성공은 독립 자랑이 아니라 실패 이후의 후속 기록이다
- 서비스 분위기는 무겁기보다 가볍고 편한 공감형이다

### 정체성 원칙
- 실명 사용 안 함
- 닉네임 계정 사용
- 글마다 익명 또는 닉네임 선택 가능
- 댓글 기능은 MVP에서 보류

### 커뮤니티 원칙
- 좋아요보다 공감형 리액션 사용
- 초기에는 댓글보다 공감 반응 중심
- 훈계형/비꼼형 상호작용을 초기에 최대한 차단
- 운영 초기는 활동량보다 신뢰를 우선

## 8. 구현 시 흔들리면 안 되는 제품 판단

### 왜 익명/닉네임 선택형인가
- 완전 공개는 글쓰기 장벽이 높다
- 완전 익명만 허용하면 서사가 끊긴다
- 익명과 닉네임을 함께 둬야 안전성과 연속성을 같이 가져갈 수 있다

### 왜 성공 글을 독립 게시글로 두지 않는가
- 서비스 주제가 성공 자랑으로 흐를 위험이 크다
- 실패 이후의 변화라는 구조가 더 명확하다
- 서비스 정체성이 흐려지지 않는다

### 왜 좋아요 대신 공감 리액션인가
- 실패 글에 좋아요는 의미가 어색하다
- 리액션 문구 자체가 커뮤니티 문화를 만든다
- 공감형 반응이 목적과 더 잘 맞는다

### 왜 댓글은 로그인 후만 가능한가
### 왜 댓글을 MVP에서 제외하는가
- 사람들은 항상 선의로 반응하지 않는다
- 실패를 공유하는 공간에서 댓글은 쉽게 상처가 된다
- 운영 비용과 모더레이션 부담이 커진다
- 현재 목적은 토론보다 기록과 공감이다

## 15. 백엔드 상태 정책

- `ACTIVE` 사용자는 일반 기능을 사용할 수 있다.
- `RESTRICTED` 사용자는 공개 조회만 가능하며 작성, 수정, 공감, 신고, 관리자 기능은 사용할 수 없다.
- `DELETED` 사용자는 인증 이후 보호 API를 사용할 수 없다.
- 비활성 카테고리는 기존 게시글 조회에는 유지되지만 새 게시글 작성과 카테고리 변경에는 사용할 수 없다.
- 숨김 또는 삭제된 게시글은 공개 상세·검색·후속 기록 조회에서 노출하지 않는다.
- 사용자 제한·활성화 운영 이력의 대상 유형은 게시글이 아닌 `USER`로 기록한다.

## 16. 운영·검증 기준

- 목록 페이지 크기는 최대 50으로 제한한다.
- 게시글 검색어는 최대 100자, 카테고리 검색어는 최대 50자로 제한한다.
- 모든 요청은 `X-Request-ID`로 추적하며 로그에 상태 코드와 처리 시간을 남긴다.
- 공개 헬스 체크 외 Actuator 메트릭은 관리자에게만 허용한다.
- 운영 프로필에서 Swagger는 기본 비활성화한다.
- 공감·신고 동시성, Flyway, 사용자·카테고리 정책은 Testcontainers MySQL 8.4로 자동 검증한다.
- Docker Compose 실행 시 비밀번호와 JWT 시크릿은 `.env`로 주입하고 이미지나 저장소에 포함하지 않는다.
- Access Token 기본 유효시간은 15분으로 유지한다.
- Refresh Token 원문은 저장하지 않고 SHA-256 해시만 저장한다.
- Refresh Token은 매 갱신 시 회전하며 로그아웃·사용자 제한 시 폐기한다.
- 로컬 HTTP에서는 Refresh 쿠키의 `Secure`를 끄고 HTTPS 운영 환경에서는 반드시 켠다.
- 로그인·회원가입·토큰 갱신은 IP, 신고는 인증 사용자 ID 기준으로 요청 횟수를 제한한다.
- `X-Forwarded-For`는 신뢰 프록시가 기존 헤더를 제거하고 다시 설정하는 환경에서만 사용한다.
- 요청 제한 키 저장소는 최대 크기와 만료 정리 정책을 가져야 한다.
- Prometheus 내부 수집 모드는 별도 관리 포트가 외부에 공개되지 않을 때만 활성화한다.
- Grafana 대시보드와 Prometheus 경보 규칙은 Compose 프로필로 재현 가능해야 한다.
