## 2. 대표 문제 해결 사례

### 2.1 게시글 목록 N+1과 쿼리 수 회귀 방지

#### 제목

> JPA 지연 로딩으로 발생한 N+1을 제거하고 목록 SQL을 최대 3회로 고정

#### 포트폴리오 본문

게시글 목록 응답을 DTO로 변환할 때 작성자와 카테고리를 지연 로딩하면서 게시글 수만큼 추가 SQL이 실행되는 문제를 발견했습니다. 단순히 fetch join을 추가하면 페이지네이션과 후속 기록 조회 책임이 섞일 수 있어, 목록에 필요한 단건 연관은 `EntityGraph`로 함께 조회하고 후속 기록 존재 여부는 현재 페이지의 게시글 ID를 모아 한 번에 조회했습니다. Hibernate Statistics 기반 통합 테스트를 추가해 게시글 10건을 조회해도 목록, 전체 개수, 후속 기록 여부를 포함한 SQL이 최대 3회를 넘지 않도록 회귀 기준을 고정했습니다.

#### 짧은 구문

> DTO 변환 과정의 지연 로딩으로 발생한 N+1을 `EntityGraph`와 페이지 단위 일괄 조회로 제거하고, 데이터 건수와 무관하게 목록 SQL이 최대 3회를 넘지 않도록 자동 테스트로 고정했습니다.

#### 핵심 역량

- JPA 영속성 컨텍스트와 지연 로딩 이해
- 페이지네이션을 보존한 조회 최적화
- 성능 개선을 쿼리 수 회귀 테스트로 유지

#### 한계

쿼리 수 감소가 모든 환경의 응답 시간 개선을 보장하지는 않는다. 데이터 분포, DB 부하와 네트워크가 포함된 실제 운영 환경에서는 별도 측정이 필요하다.

### 2.2 MySQL FULLTEXT 검색 성능 개선

#### 제목

> 5만 건 한국어 본문 검색 평균 72.47ms를 6.57ms로 단축

#### 포트폴리오 본문

제목과 본문의 `%LIKE%` 검색은 데이터가 증가할수록 노출 가능한 게시글 전체를 스캔해 일반 B-Tree 인덱스의 효과를 얻기 어려웠습니다. 별도 검색 서버는 현재 규모에 비해 운영 복잡도가 크다고 판단해 MySQL ngram FULLTEXT를 선택했습니다. FULLTEXT로 게시글 ID 페이지를 먼저 구한 뒤 `EntityGraph`로 필요한 연관 데이터를 조회하고, 한 글자 검색은 기존 LIKE로 폴백하도록 구성했습니다. MySQL 8.4에서 동일 데이터와 결과 건수를 기준으로 3회 워밍업 후 10회 측정한 결과, 5만 건의 본문 검색 평균이 72.47ms에서 6.57ms로 약 90.9% 감소했고 p95는 105.49ms에서 7.84ms로 줄었습니다.

#### 짧은 구문

> 전체 스캔이 발생하던 `%LIKE%` 검색을 MySQL ngram FULLTEXT로 전환해, 5만 건 본문 검색 평균을 72.47ms에서 6.57ms로 약 90.9% 단축했습니다.

#### 선택 근거

- 현재 데이터 규모에서는 MySQL만으로 충분한 개선을 얻을 수 있다.
- Elasticsearch 도입에 따른 별도 인프라, 동기화와 장애 지점을 피할 수 있다.
- Flyway로 인덱스를 버전 관리해 환경별 재현성을 유지할 수 있다.

#### 한계

ngram 인덱스는 저장 공간과 쓰기 비용이 증가하고 한 글자 검색, 동의어, 오탈자와 형태소 검색에는 한계가 있다. 검색 품질 요구가 커질 때 별도 검색 엔진을 검토한다.

### 2.3 공감·신고 동시성 정합성과 교착 상태

#### 제목

> lost update와 InnoDB 교착 상태를 원자적 갱신과 잠금 순서 통일로 해결

#### 포트폴리오 본문

공감과 신고 요청이 동시에 들어오면 엔티티의 카운터를 읽고 증가시킨 뒤 저장하는 방식에서 마지막 쓰기만 남는 lost update 가능성이 있었습니다. 카운터를 `count = count ± 1` 형태의 DB 원자적 UPDATE로 변경하고, 벌크 갱신 뒤 영속성 컨텍스트를 초기화해 오래된 값을 다시 읽지 않도록 했습니다. 실제 MySQL 자동 테스트로 전환하는 과정에서는 자식 행 INSERT와 부모 카운터 UPDATE의 잠금 순서가 요청마다 달라 InnoDB 교착 상태가 발생했습니다. 모든 경로에서 부모 행을 먼저 잠근 뒤 자식 행을 변경하도록 순서를 통일했습니다. Testcontainers MySQL 8.4에서 공감과 신고를 각각 8개 동시 요청으로 실행해 카운터와 실제 행 개수가 모두 8로 일치하는지 반복 검증했습니다.

#### 짧은 구문

> 공감·신고 집계의 lost update를 DB 원자적 UPDATE로 방지하고, 부모·자식 행의 잠금 순서를 통일해 InnoDB 교착 상태를 해결했습니다. MySQL 8.4에서 각각 8개 동시 요청의 카운터와 실제 행 개수 일치를 자동 검증했습니다.

#### 핵심 역량

- 애플리케이션 검증과 DB 제약의 역할 분리
- JPA 벌크 연산과 영속성 컨텍스트 불일치 처리
- 실제 DB 잠금 동작을 이용한 동시성 테스트

#### 한계

8개 동시 요청 검증은 정합성 회귀 테스트이며 장시간 고부하 성능 시험은 아니다. 운영 규모가 커지면 부하 테스트와 교착 재시도 정책을 추가로 검증해야 한다.

### 2.4 JWT 인증 수명주기와 즉시 권한 회수

#### 제목

> 서명 검증을 넘어 회전·재사용 탐지·권한 회수까지 포함한 JWT 수명주기 설계

#### 포트폴리오 본문

장기 Access Token만 사용하는 구조에서는 토큰 탈취와 관리자 제한 이후의 권한 회수 범위가 불명확했습니다. Access Token 만료를 15분으로 줄이고 Refresh Token 원문 대신 SHA-256 해시를 저장했습니다. 갱신할 때마다 Refresh Token을 회전하며 이미 사용된 토큰이 다시 들어오면 같은 계열을 모두 폐기하도록 구성했습니다. 또한 관리자 API는 JWT role claim만 신뢰하지 않고 DB의 현재 역할과 `ACTIVE` 상태를 다시 확인해 기존 토큰을 가진 제한 관리자도 즉시 차단했습니다. 잘못된 issuer 거부, 동시 갱신 시 한 요청만 성공, 로그아웃과 사용자 제한 후 세션 폐기를 통합 테스트와 HTTPS 운영 스모크로 검증했습니다.

#### 짧은 구문

> 15분 Access Token과 회전형 Refresh Token을 설계하고, 해시 저장·재사용 탐지·토큰 계열 폐기를 구현했습니다. 관리자 권한은 DB의 현재 상태를 재검증해 기존 JWT가 있어도 제한 즉시 `403`으로 차단했습니다.

#### 트레이드오프

완전한 stateless 인증보다 DB 조회와 저장 비용이 늘지만, 강제 로그아웃과 권한 회수가 필요한 사용자 콘텐츠 서비스의 보안 요구를 우선했다.

#### 한계

단일 서버와 현재 트래픽을 기준으로 한 설계다. 다중 리전이나 대규모 인증 트래픽에서는 토큰 저장소와 폐기 전파 전략을 별도로 설계해야 한다.

### 2.5 무제한 캐시 키 증가 방지

#### 제목

> 캐시 적중률보다 키 공간을 먼저 검토해 사용자 입력 기반 무제한 캐시 제거

#### 포트폴리오 본문

공개 카테고리 검색어 전체를 로컬 캐시 키로 사용하면 사용자가 고유 검색어를 계속 입력할 때 캐시 엔트리가 제한 없이 증가할 수 있었습니다. 카테고리 데이터는 작고 변경이 적다는 특성을 기준으로 검색 결과 캐시를 제거하고 전체 활성 카테고리 목록 하나만 캐시한 뒤 애플리케이션에서 필터링하도록 변경했습니다. 관리자 변경 시 해당 단일 캐시를 무효화하고, 서로 다른 검색어를 호출해도 엔트리가 하나만 유지되는지 자동 테스트로 검증했습니다.

#### 짧은 구문

> 사용자 검색어를 그대로 캐시 키로 사용해 메모리가 무제한 증가할 수 있는 구조를 발견했습니다. 전체 활성 목록만 단일 키로 캐시하고 검색은 메모리에서 수행해 캐시 엔트리를 1개로 제한했습니다.

#### 핵심 교훈

캐시는 항상 성능 개선이 아니다. 데이터 크기, 키 공간, 최대 크기, TTL과 무효화 비용을 함께 검토해야 한다.

### 2.6 관리자 운영 지표 집계 최적화

#### 제목

> 관리자 지표 API의 SQL을 7회에서 2회로 축소

#### 포트폴리오 본문

관리자 대시보드는 게시글, 후속 기록과 신고 지표를 각각 조회해 권한 확인을 포함해 총 7개의 SQL을 순차 실행했습니다. 여러 번의 DB 왕복을 줄이기 위해 여섯 개 집계를 스칼라 서브쿼리 기반의 단일 조회로 통합했고, 즉시 권한 회수를 위한 관리자 현재 상태 조회는 유지했습니다. 결과적으로 API SQL을 7회에서 2회로 줄였으며, MySQL 8.4에서 게시글 1만 건, 후속 기록 1천 건, 신고 200건을 기준으로 집계 평균이 6.75ms에서 3.18ms로 감소했습니다. 응답 값과 최대 SQL 수를 함께 확인하는 통합 테스트를 추가했습니다.

#### 짧은 구문

> 독립적으로 실행되던 관리자 지표 집계를 단일 SQL로 통합해 API 쿼리를 7회에서 2회로 줄이고, 로컬 MySQL 8.4 기준 평균 집계 시간을 6.75ms에서 3.18ms로 단축했습니다.

#### 선택하지 않은 대안

현재 평균이 3.18ms이고 관리자만 사용하는 API라 TTL 캐시는 도입하지 않았다. 캐시가 만드는 데이터 지연과 모든 쓰기 경로의 무효화 복잡도가 현재 이점보다 크다고 판단했다.

### 2.7 HTTPS 운영 경계와 배포 검증

#### 제목

> 개발 Compose를 Caddy HTTPS 단일 진입점의 운영 경계로 분리

#### 포트폴리오 본문

개발 Compose에서는 MySQL, Spring Boot와 Next.js가 각각 호스트 포트에 노출되어 있었고, 실제 HTTPS 프록시 환경에서 Secure Cookie와 동일 출처 API가 동작하는지 검증하지 못했습니다. 포트폴리오용 단일 서버에 Kubernetes를 추가하는 대신 Caddy를 유일한 공개 진입점으로 두고 기존 Compose에 운영 override를 합성했습니다. MySQL, 백엔드, 프론트엔드와 관측성 포트는 내부 네트워크로 제한하고 HTTP 리다이렉트, 보안 헤더, 운영 Swagger 비노출과 전달 IP 신뢰 경계를 구성했습니다. 별도 Docker 프로젝트를 띄우는 10단계 스모크 테스트로 회원가입, 로그인, Secure Refresh Cookie 회전, 게시글 작성, 로그아웃 후 토큰 폐기와 Caddy 외 포트 비노출을 GitHub Actions에서 자동 검증했습니다.

#### 짧은 구문

> Caddy를 HTTPS 단일 진입점으로 두고 DB·백엔드·프론트엔드 포트를 내부화했습니다. 구성 파일 검토에 그치지 않고 인증 수명주기와 포트 경계를 포함한 10단계 운영 스모크를 CI에서 검증했습니다.

#### 한계

로컬 스모크는 Caddy 내부 CA를 사용한다. 공개 도메인의 인증서, 방화벽, 외부 백업과 실제 복구 시간은 외부 서버 배포 후 별도로 검증해야 한다.

### 2.8 H2 검증을 실제 MySQL 자동 테스트로 확장

#### 제목

> H2가 보장하지 못하는 InnoDB 잠금과 Flyway 호환성을 Testcontainers로 검증

#### 포트폴리오 본문

H2의 MySQL 호환 모드는 빠른 기능 테스트에는 유용하지만 InnoDB 잠금, MySQL 고유 문법과 실제 Flyway 마이그레이션을 보장하지 못했습니다. 빠른 일반 통합 테스트는 H2로 유지하고, 동시성·제약·마이그레이션·상태 정책처럼 DB 의존성이 큰 핵심 흐름만 Testcontainers MySQL 8.4로 검증하는 계층형 전략을 선택했습니다. 이를 통해 수동으로 확인하던 동시 요청 정합성과 Flyway 적용을 Gradle 및 GitHub Actions에서 반복 가능한 테스트로 전환했습니다.

#### 짧은 구문

> H2는 빠른 기능 회귀에 유지하고, InnoDB 잠금·Flyway·유니크 제약이 중요한 흐름은 Testcontainers MySQL 8.4로 분리해 테스트 속도와 운영 DB 신뢰성의 균형을 맞췄습니다.

## 3. 포트폴리오 요약 문장

### 3.1 한 줄 소개

> 성공 중심 SNS의 비교 피로를 문제로 정의하고, 실패 기록과 후속 성장 과정을 안전하게 공유하는 서비스 Re:Fail을 기획·구현했습니다.

### 3.2 백엔드 한 문단

> Java 21과 Spring Boot 3.5.3 기반으로 도메인 패키지 경계를 나눈 단일 서버에서 JWT 인증, 게시글·후속 기록 CRUD, 공감, 신고와 관리자 moderation을 구현했습니다. JPA N+1과 검색 전체 스캔, 동시 카운터 lost update, 토큰 재사용과 권한 회수 문제를 실제 MySQL 기반 테스트로 개선했으며, Caddy HTTPS 단일 진입점과 운영 스모크를 GitHub Actions에 연결해 실행 환경까지 재현 가능하게 만들었습니다.

### 3.3 핵심 성과 세 줄

> MySQL ngram FULLTEXT를 적용해 5만 건 본문 검색 평균을 72.47ms에서 6.57ms로 줄였습니다.
>
> 원자적 UPDATE와 잠금 순서 통일로 공감·신고 동시 요청의 카운터와 실제 행 정합성을 확보했습니다.
>
> 회전형 Refresh Token과 HTTPS 단일 진입점을 구성하고 인증·포트 경계를 10단계 CI 스모크로 검증했습니다.

## 4. 작성 원칙

1. 기술 이름보다 먼저 어떤 사용자·데이터·운영 문제가 있었는지 쓴다.
2. `성능을 개선했다` 대신 데이터 수, 환경, 워밍업과 반복 횟수를 함께 쓴다.
3. `동시성을 해결했다` 대신 어떤 경쟁 조건과 잠금 순서였는지 쓴다.
4. 선택하지 않은 대안과 현재 규모에서의 판단 근거를 포함한다.
5. 자동 테스트 이름이나 재현 명령으로 검증 근거를 연결한다.
6. 로컬 측정을 운영 성과처럼 표현하지 않는다.
7. 구현하지 않은 외부 배포, 장시간 부하와 다중 서버 안정성을 완료했다고 쓰지 않는다.
