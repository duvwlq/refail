# Re:Fail 3~5분 시연 영상 대본

## 0:00~0:30 문제와 서비스

화면: README 상단 또는 메인 화면

> SNS에서는 성공과 행복이 주로 공유되지만 모든 일상이 성공적이지는 않습니다. Re:Fail은 불행을 전시하는 서비스가 아니라 실패를 돌아보고 다음 선택과 후속 기록을 남기는 서비스입니다.

## 0:30~1:20 핵심 사용자 경험

화면: 메인 목록 → 검색·필터 → 게시글 상세

> 사용자는 다이어트 같은 작은 걸림돌부터 취업이나 프로젝트 같은 큰 실패까지 기록할 수 있습니다. 글마다 익명과 닉네임을 선택하고, 댓글 대신 제한된 공감 반응만 제공합니다. 공격적인 상호작용을 줄이기 위한 MVP 정책입니다.

화면: 글 작성

> 마크다운 미리보기와 자동 임시 저장을 제공하고, 회고 가이드와 선택 항목인 다음 시도 계획을 함께 기록합니다.

## 1:20~1:50 실패에서 성장으로

화면: 후속 기록이 있는 상세 화면 또는 내 기록 보관함

> 성공 글은 별도 게시물로 만들지 않았습니다. 최초 실패 글에 다시 시도 중, 잠시 멈춤, 조금씩 나아지는 중, 극복함 상태를 연결해 실패에서 성장으로 이어지는 흐름을 완성했습니다.

## 1:50~2:30 안전과 관리자 기능

화면: 신고 → 관리자 화면 → 숨김·복구

> 사용자 콘텐츠 서비스에 필요한 신고, 숨김, 복구와 감사 이력을 구현했습니다. 제한된 사용자의 기존 JWT 접근과 숨김·삭제 게시글의 하위 기록 노출도 서버에서 차단합니다.

## 2:30~3:20 백엔드 문제 해결

화면: `PERFORMANCE.md`, 테스트 또는 아키텍처 문서

> 공감과 신고 동시 요청은 단순 read-modify-write에서 lost update가 발생할 수 있어 DB 원자적 UPDATE와 잠금 순서 통일로 해결했습니다. 실제 MySQL 8.4에서 각각 8개 동시 요청 후 행 수와 집계 값이 일치하는지 자동 검증합니다.

> 검색은 5만 건 기준 본문 LIKE 평균 72.47밀리초를 ngram FULLTEXT 6.57밀리초로 줄였습니다. 관리자 지표는 SQL 7개를 2개로 줄이고 평균 6.75밀리초를 3.18밀리초로 개선했습니다.

## 3:20~4:00 인증과 운영

화면: `ARCHITECTURE.md` JWT 흐름 → Grafana

> 인증은 15분 Access Token과 회전형 Refresh Token을 사용합니다. 서버에는 Refresh Token 해시만 저장하고, 폐기된 토큰 재사용 시 같은 계열을 모두 폐기합니다.

> 운영에서는 Prometheus와 Grafana로 요청량, 오류율, p95 지연, JVM, DB 풀, 요청 제한을 확인합니다. 관리 포트는 Docker 내부에만 공개합니다.

## 4:00~4:20 마무리

화면: GitHub Actions 또는 README 핵심 성과

> Spring Boot 단일 서버로 제품 복잡도에 맞추되 도메인 경계를 나눴고, 백엔드 40개 테스트와 프론트엔드 lint·production build를 CI에서 검증합니다. Re:Fail은 기능 수보다 정책 선택, 데이터 정합성, 측정 기반 개선을 보여주는 프로젝트입니다.

## 촬영 체크리스트

- 브라우저 확대율과 해상도를 고정한다.
- 시연 데이터와 계정을 촬영 전에 재생성한다.
- 개인 `.env`, 토큰, 쿠키, 실제 이메일이 화면에 나오지 않게 한다.
- 긴 로딩과 명령 출력은 편집하고 최종 길이를 3~5분으로 맞춘다.
- 성능 수치를 말할 때 데이터 5만 건과 MySQL 8.4 측정 조건을 함께 보여준다.
- 영상 링크가 준비되면 README 상단 실제 화면 아래에 추가한다.
