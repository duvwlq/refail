- 인증 토큰의 발급부터 폐기까지 전체 수명주기를 안전하게 관리한다.

- 구현 결과를 시연과 포트폴리오에서 재사용할 수 있는 근거로 남긴다.

## 4. 1단계: JWT Refresh Token과 로그아웃

### 목표

Access Token의 유효시간을 줄이고 Refresh Token 회전과 폐기를 도입해 탈취·재사용 위험을 낮춘다.

### 정책 결정

- Access Token은 짧은 시간만 유효한 JWT로 유지한다.
- Refresh Token은 예측 불가능한 난수로 발급한다.
- 서버에는 Refresh Token 원문이 아니라 해시만 저장한다.
- Refresh 요청마다 기존 토큰을 폐기하고 새 토큰을 발급한다.
- 폐기된 Refresh Token이 다시 사용되면 같은 토큰 계열 전체를 폐기한다.
- 로그아웃, 사용자 제한, 사용자 탈퇴 시 해당 사용자의 활성 Refresh Token을 모두 폐기한다.
- Refresh Token은 `HttpOnly`, `Secure`, `SameSite` 정책이 적용된 쿠키로 전달한다.
- Access Token은 기존과 같이 인증 헤더에 사용한다.

### 백엔드 작업

1. Refresh Token 엔티티와 저장소를 설계한다.
2. Flyway `V6` 마이그레이션을 작성한다.
3. 로그인 응답에 Access Token 발급과 Refresh Token 쿠키 설정을 연결한다.
4. `POST /api/v1/auth/refresh`를 구현한다.
5. `POST /api/v1/auth/logout`을 구현한다.
6. 토큰 회전과 재사용 탐지 로직을 구현한다.
7. 사용자 제한·활성화·탈퇴 정책과 토큰 폐기를 연결한다.
8. 만료 토큰 정리 기준과 인덱스를 정의한다.

### 프론트엔드 작업

1. API 요청에 쿠키를 포함하도록 설정한다.
2. Access Token 만료로 `401`이 발생하면 한 번만 갱신 후 원래 요청을 재시도한다.
3. 여러 요청이 동시에 `401`을 받을 때 Refresh 요청은 하나만 실행한다.
4. Refresh 실패 시 인증 상태를 제거하고 로그인 화면으로 이동한다.
5. 로그아웃 API와 화면 동작을 연결한다.

### 테스트

- 로그인 시 Access Token과 Refresh Token이 발급된다.
- 유효한 Refresh Token으로 Access Token을 재발급할 수 있다.
- 회전 후 이전 Refresh Token은 사용할 수 없다.
- 이전 토큰 재사용 시 같은 계열 토큰이 모두 폐기된다.
- 만료·위조·폐기 토큰은 `401`을 반환한다.
- `RESTRICTED`, `DELETED` 사용자는 갱신할 수 없다.
- 로그아웃 후 Refresh Token을 사용할 수 없다.
- 동시에 여러 갱신 요청이 와도 하나의 유효한 토큰 흐름만 남는다.
- 실제 MySQL Testcontainers에서 유니크 제약과 동시 갱신을 검증한다.

### 완료 조건

- Access Token 만료 후 사용자가 정상적으로 세션을 이어갈 수 있다.
- 로그아웃과 사용자 제한 후 기존 Refresh Token을 사용할 수 없다.
- 토큰 원문이 데이터베이스와 로그에 저장되지 않는다.
- API 명세와 보안 정책 문서가 갱신된다.

### 산출물

- 인증 수명주기 코드와 테스트
- Flyway 마이그레이션
- JWT 발급·회전·폐기 흐름도
- 보안 선택 근거와 남은 위험 문서

## 5. 2단계: 검색 성능 측정과 FULLTEXT 검토

### 목표

현재 제목·본문 `%LIKE%` 검색의 한계를 실제 데이터로 확인하고 FULLTEXT 적용 여부를 결정한다.

### 작업

1. 1만 건과 5만 건 게시글을 만드는 재현 가능한 성능 데이터 생성기를 작성한다.
2. 제목 검색, 본문 검색, 카테고리 결합 검색의 기준 응답 시간을 측정한다.
3. `EXPLAIN ANALYZE`로 실행 계획과 스캔 행 수를 기록한다.
4. MySQL FULLTEXT 인덱스를 별도 Flyway 마이그레이션 후보로 작성한다.
5. 자연어 검색과 Boolean Mode의 한글 검색 품질을 비교한다.
6. FULLTEXT가 적합하지 않으면 `%LIKE%` 유지와 전환 조건을 문서화한다.
7. 적용 시 기존 필터·정렬·페이지네이션과 결합한다.

### 측정 항목

- 데이터 건수
- 평균, p95 응답 시간
- 스캔 행 수
- 반환 결과 정확도
- 인덱스 크기
- 게시글 INSERT 비용 변화

### 완료 조건

- 개선 전후 결과가 같은 조건으로 측정되어 있다.
- FULLTEXT 도입 또는 보류 결정의 근거가 문서화되어 있다.
- 검색 결과와 공개 상태 정책이 회귀 테스트로 보호된다.
- 검증되지 않은 성능 수치를 README에 사용하지 않는다.

## 6. 3단계: 관리자 운영 지표 최적화

### 목표

운영 지표 API의 여러 집계 쿼리를 측정하고 필요한 범위에서 쿼리 수와 응답 시간을 줄인다.

### 작업

1. 현재 운영 지표 API의 SQL 수와 응답 시간을 회귀 테스트로 고정한다.
2. 조건부 집계 또는 전용 조회 DTO로 쿼리 통합 가능성을 검토한다.
3. 인덱스 사용 여부를 실제 MySQL 실행 계획으로 확인한다.
4. 조회 빈도와 데이터 변경 빈도를 측정한다.
5. 필요할 때만 짧은 TTL 로컬 캐시를 적용한다.
6. 관리자 조치 후 캐시 무효화가 필요한지 결정한다.

### 완료 조건

- 개선 전후 SQL 수와 응답 시간이 기록되어 있다.
- 지표 값이 기존 결과와 동일하다.
- 캐시 적용 시 최대 지연 허용 시간과 무효화 정책이 문서화되어 있다.

## 7. 4단계: 주요 API 요청 제한

### 목표

계정 공격과 반복 신고 같은 과도한 요청을 애플리케이션 수준에서 제한한다.

### 우선 적용 API

- 로그인
- 회원가입
- 토큰 갱신
- 신고 생성

### 작업

1. 사용자 ID와 IP 중 어떤 식별자를 사용할지 API별로 결정한다.
2. 허용 횟수와 시간 구간을 환경 변수로 분리한다.
3. 단일 서버 기준 인메모리 제한기를 우선 구현한다.
4. 제한 초과 시 `429 Too Many Requests`와 `Retry-After`를 반환한다.
5. 프록시 환경의 신뢰할 수 있는 클라이언트 IP 처리 정책을 정의한다.
6. 제한 발생 횟수를 Micrometer 메트릭으로 기록한다.
7. 다중 서버 전환 시 Redis 기반 제한기로 교체할 조건을 문서화한다.

### 완료 조건

- 정상 사용자는 제한에 영향을 받지 않는다.
- 경계값 이후 요청은 DB와 인증 로직까지 전달되지 않는다.
- 제한 우회와 메모리 무제한 증가를 방지하는 테스트가 있다.
- 제한 기준과 사용자 안내가 API 명세에 포함된다.

## 8. 5단계: 운영 대시보드

### 목표

현재 제공 중인 Actuator·Prometheus 메트릭을 운영자가 빠르게 판단할 수 있는 화면으로 구성한다.

### 작업

1. Docker Compose에 선택형 Prometheus·Grafana 프로필을 추가한다.
2. 다음 지표를 대시보드로 구성한다.
   - 요청 수
   - 4xx·5xx 오류율
   - 평균·p95 응답 시간
   - JVM 메모리와 GC
   - DB 커넥션 풀
   - 요청 제한 발생 수
3. 대시보드 설정을 저장소에서 재현 가능하게 관리한다.
4. 오류율과 응답 시간의 경고 기준을 정의한다.
5. 토큰, 게시글 본문, 개인정보가 메트릭 레이블에 포함되지 않는지 확인한다.

### 완료 조건

- Compose 프로필 하나로 대시보드를 실행할 수 있다.
- 장애 요청을 Request ID와 메트릭으로 추적하는 시나리오가 문서화되어 있다.
- 대시보드 화면을 포트폴리오 자료에 사용할 수 있다.

## 11. 작업 중단 기준

다음 조건에 해당하면 구현을 확대하지 않고 측정과 문서화에서 종료한다.

- FULLTEXT가 현재 데이터 규모에서 의미 있는 개선을 만들지 못한다.
- 관리자 지표가 이미 충분히 빠르고 호출 빈도가 낮아 캐시 이점이 없다.
- 단일 서버에서 Redis 도입 비용이 얻는 이점보다 크다.
- 운영 대시보드가 포트폴리오 목적보다 과도한 관리 비용을 만든다.

## 12. 바로 시작할 작업

첫 작업은 `JWT Refresh Token과 로그아웃`이다.

구현 순서는 다음과 같다.

1. 현재 로그인·JWT·프론트엔드 인증 저장 방식 확인
2. 토큰 정책과 API 계약 확정
3. 실패하는 인증 통합 테스트 작성
4. Flyway `V6`와 Refresh Token 저장 구조 구현
5. 토큰 발급·회전·폐기 서비스 구현
6. 프론트엔드 자동 갱신과 로그아웃 연결
7. 실제 MySQL 동시 갱신 테스트
8. 전체 회귀 테스트와 문서 갱신
