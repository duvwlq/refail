# Re:Fail 면접 답변

## 사용 방법

- 30초 답변은 결론과 핵심 근거만 말한다.
- 1분 답변은 문제, 선택, 구현, 검증과 한계 순서로 확장한다.
- 수치는 로컬 Docker Desktop과 MySQL 8.4에서 측정한 결과임을 함께 밝힌다.

---

## 1. 왜 이 서비스를 만들었나요?

### 30초 답변

성공과 행복이 주로 노출되는 SNS 환경에서는 자신의 실패를 더 크게 느끼고 타인과 비교하기 쉽다고 생각했습니다. 그래서 실패를 불행으로 전시하는 대신 원인과 다음 선택을 기록하고, 후속 기록으로 다시 시도하거나 극복하는 과정까지 연결하는 Re:Fail을 만들었습니다.

### 1분 답변

문제의 출발점은 실제 삶에는 실패가 많지만 기존 SNS에서는 주로 성공 결과만 보인다는 점이었습니다. 이런 환경에서는 실패를 학습 과정이 아니라 개인의 가치 문제로 받아들이기 쉽습니다. Re:Fail은 사용자가 크고 작은 실패를 익명 또는 닉네임으로 기록하고, 공격적인 댓글 대신 제한된 공감 반응을 받도록 설계했습니다. 이후 `다시 시도 중`, `잠시 멈춤`, `조금씩 나아지는 중`, `극복함` 상태의 후속 기록을 연결했습니다. 핵심은 실패를 모으는 것이 아니라 같은 실수를 반복하지 않도록 회고와 다음 선택을 남기는 것입니다.

## 2. 왜 완전 익명이나 일반 댓글을 선택하지 않았나요?

### 30초 답변

완전 익명은 글쓰기 장벽은 낮지만 본인의 기록 관리와 성장 흐름 연결이 어렵고, 실명 중심 구조는 실패를 쓰는 부담이 큽니다. 그래서 외부 공개는 익명·닉네임 중 선택하게 하고 내부 소유권은 계정으로 관리했습니다. 댓글은 조롱과 과도한 충고의 위험 때문에 MVP에서 제외하고 제한된 공감과 신고 흐름을 우선했습니다.

### 1분 답변

실패 공유에는 감정적 안전과 기록의 연속성이라는 상충하는 요구가 있습니다. 완전 익명만 사용하면 부담은 낮지만 같은 사용자가 후속 기록을 이어 쓰거나 익명 글을 다시 관리하기 어렵습니다. 반대로 실명이나 강한 공개 구조는 무거운 실패를 쓰지 못하게 할 수 있습니다. 그래서 게시글의 외부 공개 방식은 익명과 닉네임 중 선택하게 하고, 서버는 JWT 사용자 ID로 소유권을 검증합니다. 댓글은 상호작용을 늘리지만 취약한 경험에 대한 조롱과 운영 부담도 크게 만듭니다. 실제 사용자와 moderation 역량을 검증하기 전에는 정해진 공감 반응과 신고·숨김 흐름이 서비스 목적에 더 적합하다고 판단했습니다.

## 3. 가장 어려웠던 백엔드 문제는 무엇이었나요?

### 30초 답변

가장 어려웠던 문제는 공감·신고 동시 요청의 정합성이었습니다. lost update를 원자적 UPDATE로 막은 뒤 실제 MySQL 테스트에서 InnoDB 교착이 새로 드러났고, 부모 카운터와 자식 행의 잠금 순서를 통일해 해결했습니다. 각각 8개 동시 요청에서 집계와 실제 행 개수 일치를 자동 검증했습니다.

### 1분 답변

기능 자체보다 동시 요청에서도 집계가 실제 데이터와 일치하게 만드는 과정이 가장 어려웠습니다. 처음에는 엔티티 카운터를 읽고 증가시킨 뒤 저장해 동시에 실행되면 마지막 쓰기만 남을 가능성이 있었습니다. 이를 DB의 `count = count ± 1` 원자적 UPDATE로 변경했습니다. 그런데 H2에서는 보이지 않던 InnoDB 교착 상태가 Testcontainers MySQL 8.4에서 발생했습니다. 자식 공감·신고 행과 부모 게시글 카운터를 변경하는 순서가 요청 경로마다 달랐기 때문입니다. 모든 경로에서 잠금 순서를 통일한 뒤 공감·신고 각각 8개 동시 요청에서 카운터와 실제 행이 모두 8인지 검증했습니다. 이 경험으로 정합성 문제는 애플리케이션 코드뿐 아니라 실제 DB 잠금 순서까지 봐야 한다는 점을 확인했습니다.

## 4. 동시성 문제를 어떻게 재현하고 해결했나요?

### 30초 답변

서로 다른 사용자 8명이 같은 게시글에 동시에 공감하거나 신고하도록 실행 장벽을 둔 통합 테스트를 만들었습니다. 응답 성공 여부만 보지 않고 부모 카운터와 자식 테이블의 실제 행 수를 함께 비교했습니다. 원자적 UPDATE와 잠금 순서 통일 후 두 값이 모두 8로 일치하도록 회귀 테스트를 고정했습니다.

### 1분 답변

재현 단계에서는 하나의 게시글과 서로 다른 사용자 8명을 준비하고 스레드들이 같은 시점에 시작하도록 동기화했습니다. 공감과 신고는 사용자·게시글 조합에 유니크 제약이 있으므로 각 사용자는 한 번만 요청했습니다. 검증은 HTTP나 서비스 호출의 성공 개수에 그치지 않고 게시글의 집계 카운터와 공감·신고 테이블의 실제 행 개수를 함께 확인했습니다. 해결 과정에서 카운터는 DB 원자적 UPDATE로 바꾸고 벌크 갱신 후 영속성 컨텍스트를 초기화했습니다. Testcontainers에서 발견한 교착은 부모와 자식 행의 변경 순서를 모든 경로에서 동일하게 만들어 제거했습니다. 이 테스트는 정합성 회귀 검증이며 처리량이나 최대 동시 사용자 수를 측정하는 부하 테스트는 아닙니다.

## 5. FULLTEXT를 선택하고 Elasticsearch를 도입하지 않은 이유는 무엇인가요?

### 30초 답변

현재 요구사항은 제목·본문·감정 태그의 한국어 키워드 검색이고 데이터 기준선은 5만 건이었습니다. MySQL ngram FULLTEXT만으로 본문 검색 평균을 `72.47ms`에서 `6.57ms`로 줄일 수 있어, 별도 검색 서버의 인프라·동기화·장애 복잡도를 추가하지 않았습니다. 형태소·동의어 검색 요구가 생길 때 전환할 계획입니다.

### 1분 답변

기존 `%LIKE%` 검색은 노출 가능한 게시글을 전체 스캔해 5만 건에서 본문 검색 평균이 72.47ms였습니다. 대안으로 복합 B-Tree, MySQL FULLTEXT와 Elasticsearch를 비교했습니다. 앞뒤 와일드카드에는 일반 B-Tree가 적합하지 않고, Elasticsearch는 검색 품질은 높지만 별도 인프라와 데이터 동기화, 장애 지점을 추가합니다. 현재는 텍스트 중심 단일 서버 MVP이고 필요한 검색도 키워드 기반이라 MySQL ngram FULLTEXT를 선택했습니다. 같은 데이터에서 결과 건수를 맞추고 3회 워밍업 후 10회 측정해 평균 6.57ms, p95 7.84ms를 확인했습니다. 한 글자는 LIKE로 폴백하고, 향후 형태소·동의어·오탈자 검색이 제품 요구가 될 때 별도 검색 엔진을 검토합니다.

## 6. JWT인데 왜 DB 상태를 다시 조회하나요?

### 30초 답변

JWT role claim만 신뢰하면 관리자를 제한해도 기존 토큰이 만료될 때까지 관리자 기능을 사용할 수 있습니다. Re:Fail은 관리자 API와 보호된 주요 흐름에서 DB의 현재 역할과 사용자 상태를 확인해 즉시 권한을 회수합니다. stateless의 일부 장점을 포기하지만 moderation 서비스의 안전 요구를 우선했습니다.

### 1분 답변

JWT는 서명된 권한 정보를 빠르게 검증할 수 있지만 발급 이후의 상태 변경을 스스로 알 수 없습니다. 실제 리뷰에서 제한된 관리자가 기존 JWT로 관리자 지표를 조회하고 자기 계정을 다시 활성화할 수 있는 문제가 있었습니다. 이를 해결하기 위해 모든 관리자 서비스에서 토큰의 사용자 ID로 현재 DB 사용자를 조회하고 역할과 `ACTIVE` 상태를 다시 검증했습니다. 일반 사용자 제한 시에는 활성 Refresh Token도 폐기합니다. Access Token은 15분으로 짧게 유지하고 Refresh Token은 회전·재사용 탐지 구조로 관리합니다. 추가 DB 조회 비용은 있지만 관리자 요청 빈도보다 즉시 권한 회수와 감사 가능성이 더 중요하다고 판단했습니다.

## 7. 왜 모놀리식 Spring Boot를 선택했나요?

### 30초 답변

현재 서비스는 텍스트 중심 MVP이며 인증, 게시글, 공감, 신고가 하나의 트랜잭션 안에서 처리됩니다. 마이크로서비스의 독립 확장성보다 배포·네트워크·분산 트랜잭션 비용이 더 크다고 판단해 Spring Boot 단일 서버를 선택했습니다. 대신 도메인 패키지와 유스케이스 서비스로 책임 경계를 나눴습니다.

### 1분 답변

기술 구조는 제품 복잡도에 맞춰야 한다고 판단했습니다. 현재 기능은 인증, 게시글, 후속 기록, 공감, 신고와 관리자 moderation이며 독립 배포나 서로 다른 확장 주기가 필요한 근거가 없습니다. 이를 서비스별로 분리하면 네트워크 호출, 분산 트랜잭션, 배포와 관측 복잡도가 먼저 증가합니다. 그래서 하나의 Spring Boot 배포 단위를 유지하고 `auth`, `post`, `reaction`, `report`, `admin` 패키지로 경계를 나눴습니다. 리팩터링 과정에서는 조회, 변경, 관리자 지표 책임도 유스케이스 서비스로 분리했습니다. 실제 트래픽과 조직이 커져 독립 확장이 필요한 도메인이 확인될 때 분리하는 것이 비용 대비 합리적입니다.

## 8. 테스트 전략을 어떻게 구성했나요?

### 30초 답변

빠른 API·정책 회귀는 H2 통합 테스트로 유지하고, InnoDB 잠금·Flyway·MySQL 제약이 중요한 흐름은 Testcontainers MySQL 8.4로 분리했습니다. 여기에 Playwright P0 3개와 HTTPS 운영 스모크 10단계를 추가해 API, 브라우저와 배포 경계를 GitHub Actions에서 각각 검증합니다.

### 1분 답변

테스트 피라미드를 속도와 실제 환경 충실도 기준으로 나눴습니다. 입력 검증, 권한과 일반 CRUD는 H2 기반 Spring 통합 테스트로 빠르게 확인합니다. H2가 보장하지 못하는 InnoDB 잠금, FULLTEXT, Flyway와 유니크 제약은 Testcontainers MySQL 8.4에서 검증합니다. 브라우저에서는 공개 탐색, 작성자 CRUD·후속 기록, 관리자 신고·숨김·복구를 Playwright P0 3개로 구성했습니다. 마지막으로 Caddy 내부 CA를 사용하는 별도 운영형 Compose에서 HTTPS, Secure Cookie 회전, 로그아웃 폐기와 포트 비노출 10단계를 검사합니다. GitHub Actions는 네 계층을 독립 작업으로 실행하고 E2E 실패 자료를 artifact로 남깁니다.

## 9. 운영 배포에서 아직 부족한 점은 무엇인가요?

### 30초 답변

현재는 Caddy HTTPS 단일 진입점과 내부 서비스 비노출을 로컬·CI에서 검증했지만 공개 도메인과 외부 서버 배포는 완료하지 않았습니다. 실제 인증서 발급, 방화벽, 자동 외부 백업과 복구 시간, 장시간 부하·디스크 고갈 검증이 남아 있습니다.

### 1분 답변

운영형 Compose는 Caddy만 HTTP·HTTPS를 공개하고 MySQL, Spring Boot, Next.js와 관측성 포트를 내부화합니다. Secure Refresh Cookie와 인증 수명주기는 10단계 스모크로 검증했습니다. 다만 로컬에서는 Caddy 내부 CA를 사용하므로 공개 인증서 발급이나 실제 인터넷 경로를 증명하지 못했습니다. 서버 방화벽과 DNS, 외부 저장소 백업, 백업 복원 시간과 데이터 손실 범위도 아직 측정하지 않았습니다. 장시간 부하, 디스크 고갈과 다중 인스턴스 장애 역시 범위 밖입니다. 다음 단계는 비용 상한이 있는 단일 서버에 실제 도메인을 연결하고 자동 백업을 복원해 RTO와 RPO의 기준선을 기록하는 것입니다.
