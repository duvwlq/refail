## 2. 발견 사항

### [P1] 공감·신고 집계 값에 lost update가 발생할 수 있다

- 근거: `ReactionServiceImpl.java:56-65`는 반응을 저장한 뒤 조회해 둔 `Post` 객체의 카운트를 증가시키고, `ReportServiceImpl.java:39-54`도 같은 방식으로 신고 수를 증가시킨다. `Post.java:68-72`, `Post.java:115-126`에는 일반 정수 필드와 증감 메서드만 있고 `@Version`, 잠금, 원자적 UPDATE가 없다.
- 문제 상황: 서로 다른 사용자 두 명이 반응 수가 0인 같은 게시글에 동시에 공감한다. 두 트랜잭션 모두 0을 읽고 각각 1로 저장할 수 있다. 반응 행은 2개지만 `posts.reaction_count`는 1이 된다. 신고 수도 같은 문제가 발생한다.
- 영향: 데이터 정합성, 인기순 정렬, 게시글 응답, 운영 지표의 신뢰도가 깨진다.
- 재현 방법: 실제 MySQL에서 두 개의 독립 트랜잭션 또는 `CountDownLatch` 기반 동시성 테스트로 서로 다른 사용자 두 명의 공감 등록을 동시에 실행한 뒤 `reactions` 행 개수와 `posts.reaction_count`를 비교한다.
- 권장 수정: 반응 행 생성·삭제 성공 후 `UPDATE posts SET reaction_count = reaction_count + 1 WHERE id = ?` 형태의 원자적 증감 쿼리를 사용한다. 신고 수도 같은 방식으로 처리한다. 동일 사용자 중복 요청은 유니크 제약 위반을 도메인 충돌 응답으로 변환한다.
- 필요한 테스트: 서로 다른 사용자 20명의 동시 공감 등록·삭제, 같은 사용자의 동시 등록, 동시 신고 후 행 개수와 집계 값 일치 테스트를 실제 MySQL에서 수행한다.
- 포트폴리오 가치: JPA 영속성 컨텍스트만으로 해결되지 않는 동시성 문제와 원자적 데이터 갱신을 설명할 수 있다.
- 예상 작업량: M

### [P1] 숨김·삭제 게시글의 후속 기록이 공개 API로 노출된다

- 근거: `PostController.java:157-160`의 공개 API는 `PostService.getUpdates`를 직접 호출한다. `PostServiceImpl.java:199-203`은 부모 게시글의 삭제·숨김 상태를 검사하지 않고 `post_updates`만 조회한다. 반면 상세 조회는 `PostServiceImpl.java:220-225`에서 숨김·삭제 상태를 검사한다.
- 문제 상황: 관리자가 신고된 게시글을 숨기거나 작성자가 게시글을 삭제해도 게시글 ID를 아는 사용자는 `/api/v1/posts/{postId}/updates`를 호출해 후속 기록 내용을 읽을 수 있다.
- 영향: moderation 우회, 삭제 기대 위반, 민감한 실패 기록 노출.
- 재현 방법: 로컬 실행 서버에서 후속 기록이 있는 게시글을 숨긴 뒤 확인했다. `GET /api/v1/posts/{id}`는 `404`, `GET /api/v1/posts/{id}/updates`는 `200`과 후속 기록 1건을 반환했다. 재현 데이터는 검증 후 숨김 해제하고 소프트 삭제했다.
- 권장 수정: 공개 후속 기록 조회 전에 부모 게시글의 노출 가능 상태를 반드시 검증한다. 상세 조회 내부에서는 같은 검증·쿼리가 중복되지 않도록 비공개 매핑 메서드를 분리한다.
- 필요한 테스트: 숨김 게시글, 삭제 게시글, 존재하지 않는 게시글의 후속 기록 조회가 모두 `404`인지 검증한다.
- 포트폴리오 가치: 소프트 삭제와 moderation 정책을 모든 하위 리소스에 일관되게 적용하는 경험을 보여준다.
- 예상 작업량: S

### [P1] 제한된 관리자가 기존 JWT로 관리자 기능을 계속 사용한다

- 근거: `JwtAuthenticationFilter.java:32-41`은 JWT의 역할 claim으로 Spring Security 권한을 만든다. `AdminServiceImpl.java:148-153`의 `getAdmin`은 현재 DB 역할만 확인하고 사용자 상태는 확인하지 않는다. `AdminServiceImpl.java:47-52`, `AdminServiceImpl.java:136-145`의 신고 조회와 지표 조회는 `getAdmin`조차 호출하지 않는다.
- 문제 상황: 관리자가 토큰을 발급받은 후 계정이 제한되어도 토큰에는 `ROLE_ADMIN`이 남는다. 기존 토큰으로 신고 사유를 조회하고 게시글을 숨기거나 자신을 다시 활성화할 수 있다.
- 영향: 권한 회수 실패, 관리자 계정 침해 대응 불가, 운영 정보 노출.
- 재현 방법: 로컬 관리자 ID 4를 기존 토큰으로 제한한 후 같은 토큰으로 `/api/v1/admin/metrics`와 `/api/v1/admin/users/4/activate`를 호출했다. 두 요청 모두 `200`이었다. 검증 후 계정은 다시 활성화했다.
- 권장 수정: 모든 관리자 서비스 메서드에서 현재 DB의 역할과 `ACTIVE` 상태를 검증한다. 신고 조회와 지표 조회에도 관리자 ID를 전달한다. 장기적으로는 토큰 버전 또는 짧은 만료 시간을 사용해 강제 로그아웃 범위를 정의한다.
- 필요한 테스트: 제한된 관리자, 역할이 변경된 관리자, 삭제된 관리자의 기존 토큰으로 모든 관리자 API가 `403`인지 검증한다.
- 포트폴리오 가치: JWT의 stateless 특성과 권한 회수 전략의 trade-off를 설명할 수 있다.
- 예상 작업량: M

### [P1] 공개 검색어별 로컬 캐시가 제한 없이 증가한다

- 근거: `CategoryServiceImpl.java:19`는 사용자 입력 검색어 전체를 키로 `activeCategories`에 캐시한다. `CacheConfig.java:15`는 크기·TTL 제한이 없는 `ConcurrentMapCacheManager`를 사용한다. `CategoryController.java:24-25`는 인증과 검색어 길이 제한 없이 공개되어 있다.
- 문제 상황: 외부 사용자가 매 요청마다 다른 검색어를 보내면 빈 결과까지 모두 힙 메모리에 영구 보관된다.
- 영향: 장시간 운영 시 메모리 증가와 최종적인 OOM 가능성.
- 재현 방법: 수만 개의 고유한 `keyword`로 카테고리 API를 호출한 뒤 `activeCategories` 캐시 크기와 JVM 힙 사용량을 관찰한다.
- 권장 수정: 변경이 적은 전체 활성 카테고리 목록만 캐시하고 검색 결과는 캐시하지 않는다. 검색 결과도 캐시해야 한다면 Caffeine의 최대 크기와 TTL을 명시한다.
- 필요한 테스트: 서로 다른 검색어 호출이 캐시 엔트리를 늘리지 않는지, 관리자 변경 후 전체 목록 캐시가 무효화되는지 검증한다.
- 포트폴리오 가치: 캐시가 항상 성능 개선이 아니며 키 공간과 만료 정책이 필요하다는 점을 설명할 수 있다.
- 예상 작업량: S

### [P1] 운영 기본 설정에 알려진 JWT 시크릿이 존재한다

- 근거: `application.yml:44`는 환경 변수가 없을 때 저장소에 공개된 문자열을 JWT 서명 키로 사용한다. `application-docker.yml:5-6`과 `compose.yaml`에도 고정 DB 계정이 있다.
- 문제 상황: 이 설정 그대로 외부에 배포하면 누구나 저장소의 시크릿으로 임의의 `ADMIN` JWT를 만들 수 있다.
- 영향: 인증 위조와 전체 관리자 권한 탈취.
- 재현 방법: 기본 시크릿으로 관리자 role claim을 포함한 토큰을 서명한 뒤 관리자 API를 호출한다. 현재 파서는 서명만 맞으면 DB에 해당 ID가 존재하는 범위에서 claim 역할을 Security 권한으로 사용한다.
- 권장 수정: 운영 기본 프로필은 `${JWT_SECRET}`처럼 값이 없으면 시작에 실패하게 한다. `local`과 `docker` 개발 프로필에만 명시적인 개발용 시크릿을 둔다. 배포 문서에 최소 길이와 주입 방식을 적는다.
- 필요한 테스트: 운영 프로필에서 시크릿 미설정 시 애플리케이션 시작 실패, 잘못된 서명과 issuer 토큰 거부 테스트.
- 포트폴리오 가치: secure default와 환경별 설정 분리를 보여준다.
- 예상 작업량: S

### [P2] 중복 생성 경쟁 조건이 일관된 API 오류로 변환되지 않는다

- 근거: `AuthServiceImpl.java:31-46`은 이메일·닉네임 존재 확인 후 저장하고, `ReportServiceImpl.java:39-53`과 `ReactionServiceImpl.java:56-64`도 조회 후 저장한다. DB 유니크 제약은 있지만 `GlobalExceptionHandler.java:48-56`은 제약 위반을 일반 `500`으로 처리한다.
- 문제 상황: 같은 이메일 회원가입이나 같은 사용자의 신고·공감 요청이 거의 동시에 들어오면 두 요청 모두 사전 조회를 통과하고 하나가 유니크 제약에서 실패한다.
- 영향: 정상적인 경쟁 요청이 서버 장애처럼 응답되고 클라이언트가 안전하게 처리하기 어렵다.
- 재현 방법: 동일 payload를 두 스레드에서 동시에 전송하고 응답 코드와 `DataIntegrityViolationException` 로그를 확인한다.
- 권장 수정: 유니크 제약을 최종 정합성 기준으로 유지하고, 제약 이름별 충돌을 `409 Conflict` 도메인 오류로 변환한다. 사전 조회는 친절한 빠른 실패 용도로만 사용한다.
- 필요한 테스트: 동일 이메일, 닉네임, 신고, 공감의 동시 요청에서 한 건만 성공하고 나머지는 정해진 `409`를 반환하는지 검증한다.
- 포트폴리오 가치: check-then-act 경쟁 조건과 DB 제약의 역할을 설명할 수 있다.
- 예상 작업량: M

### [P2] 상태 정책이 서비스별로 일관되지 않다

- 근거: `PostServiceImpl.java:206-211`, `ReactionServiceImpl.java:85-90`, `ReportServiceImpl.java:58-63`은 `RESTRICTED`만 거부하고 `UserStatus.DELETED`는 거부하지 않는다. `PostServiceImpl.java:215-217`은 카테고리 존재만 확인해 비활성 카테고리에도 새 게시글을 작성할 수 있다.
- 문제 상황: DB에서 삭제 상태가 된 사용자가 기존 토큰으로 활동하거나, UI에서 사라진 비활성 카테고리 ID로 게시글을 작성한다.
- 영향: 상태 모델과 실제 정책 불일치, 관리 기능 우회.
- 재현 방법: 사용자를 `DELETED`로 변경한 뒤 기존 토큰으로 게시글을 작성하고, 비활성 카테고리 ID로 작성 요청을 보낸다.
- 권장 수정: 공통 활성 사용자 조회 정책을 정의해 `ACTIVE`만 허용한다. 게시글 작성·수정은 활성 카테고리만 허용하고 기존 글 조회는 비활성 카테고리도 보존한다.
- 필요한 테스트: `ACTIVE`, `RESTRICTED`, `DELETED` 사용자와 활성·비활성 카테고리 조합 테스트.
- 포트폴리오 가치: 상태 전이와 도메인 불변식을 중앙화하는 설계를 보여준다.
- 예상 작업량: S

### [P2] 운영 지표와 감사 로그의 의미가 실제 대상과 다를 수 있다

- 근거: `PostUpdateRepository.java:22-25`의 후속 기록 집계는 부모 게시글의 삭제 여부를 필터링하지 않는다. `AdminServiceImpl.java:137-145`는 삭제되지 않은 게시글 수를 분모로 사용하면서 삭제 게시글의 후속 기록을 분자에 포함할 수 있다. 사용자 제한·활성화 이력도 `AdminServiceImpl.java:89`, `AdminServiceImpl.java:100`에서 대상 유형을 `POST`로 저장하며 `ReportTargetType.java:3-4`에는 `POST`만 존재한다.
- 문제 상황: 후속 기록이 있는 게시글을 삭제하면 후속 작성률이 100%를 넘을 수 있다. 사용자 ID 10에 대한 제한 이력이 게시글 ID 10에 대한 조치로 해석된다.
- 영향: 운영 대시보드와 감사 이력을 신뢰할 수 없다.
- 재현 방법: 후속 기록이 있는 게시글을 삭제한 뒤 metrics를 조회하고, 사용자 제한 후 `moderation_actions.target_type`을 확인한다.
- 권장 수정: 지표 쿼리에 부모 게시글 노출·삭제 정책을 명시한다. moderation 대상 enum에 `USER`를 추가하고 사용자 조치에는 `USER`를 기록한다.
- 필요한 테스트: 삭제 게시글 포함 여부와 사용자 moderation 이력의 target type 검증.
- 포트폴리오 가치: 지표 정의와 감사 데이터의 의미적 정합성을 다룬 경험을 보여준다.
- 예상 작업량: S

### [P2] 공개 조회 비용과 요청 크기 제한이 불완전하다

- 근거: `PostController.java:67-70`은 일반 목록 크기만 제한하지만 `PostController.java:76-81`의 내 목록은 제한하지 않는다. 실행 서버에서 `/api/v1/posts/me?size=101`이 `200`으로 처리됐다. `PostController.java:63`, `CategoryController.java:24`의 검색어 길이 제한도 없고 게시글 검색은 `PostServiceImpl.java:86-92`의 `%keyword%` 전체 검색이다. 게시글·후속 기록·신고 사유 본문에도 명시적 최대 길이가 없다.
- 문제 상황: 매우 큰 페이지 크기, 긴 검색어, 큰 본문 요청을 반복해 DB와 힙 사용량을 높일 수 있다.
- 영향: 응답 지연과 가용성 저하.
- 재현 방법: 큰 `size`, 고유 검색어, 대용량 본문 요청을 반복하고 DB CPU·메모리·응답 시간을 측정한다.
- 권장 수정: 모든 페이지 API에 공통 페이지 검증을 적용하고 검색어와 텍스트 필드에 정책상 최대 길이를 둔다. 서버 request size와 타임아웃도 배포 환경에서 명시한다.
- 필요한 테스트: 페이지·검색어·본문의 최소·최대 경계값 테스트.
- 포트폴리오 가치: 입력 검증을 데이터 형식뿐 아니라 자원 사용량 보호로 확장한 경험을 보여준다.
- 예상 작업량: S

### [P2] H2 테스트만으로 MySQL 동시성과 Flyway를 검증한다는 근거가 부족하다

- 근거: `src/test/resources/application.yml:2`는 모든 통합 테스트에 H2 MySQL 호환 모드를 사용하고 Flyway는 비활성화한다. 전체 테스트에는 동시성 테스트가 없으며 현재 테스트 메서드는 9개다.
- 문제 상황: MySQL 격리 수준, 잠금, 인덱스, TEXT 길이, Flyway SQL 문법과 제약 이름 처리 차이가 H2에서는 드러나지 않는다.
- 영향: 테스트 통과 후 실제 MySQL에서만 장애가 발생할 수 있다.
- 재현 방법: 동일 테스트를 Testcontainers MySQL 8에서 실행해 H2 결과와 비교한다.
- 권장 수정: 빠른 일반 통합 테스트는 H2로 유지하되 동시성, 유니크 제약, Flyway, 인덱스 관련 핵심 테스트만 MySQL Testcontainers 태그로 분리한다.
- 필요한 테스트: Flyway 전체 적용, 동시 공감·신고, 중복 제약 응답, 주요 조회 `EXPLAIN` 검증.
- 포트폴리오 가치: 테스트 피라미드와 운영 DB 호환성의 현실적인 균형을 설명할 수 있다.
- 예상 작업량: L

### [P3] 운영 관측성과 인증 방어가 최소 수준이다

- 근거: 빌드에 Actuator·Micrometer가 없고 요청 추적 ID나 구조화 로그 설정도 없다. `HealthController.java:38-43`은 공개 요청마다 직접 DB 연결을 확인한다. 로그인 횟수 제한, Refresh Token, 로그아웃 무효화 정책도 없다. `JwtTokenProvider.java:62-67`은 서명과 만료는 검증하지만 설정한 issuer를 요구하지 않는다.
- 문제 상황: 오류율·지연 증가 원인을 추적하기 어렵고 로그인 brute force나 토큰 탈취 시 대응 수단이 제한된다.
- 영향: 장애 탐지와 보안 대응 시간 증가.
- 재현 방법: 요청별 로그를 연결하거나 API별 지연 백분위, 인증 실패율을 현재 시스템에서 조회할 수 있는지 확인한다.
- 권장 수정: 포트폴리오 제출 전에는 요청 ID와 핵심 보안 로그를 우선 추가한다. 실제 운영 전에는 Actuator readiness/liveness, Micrometer 지표, 로그인 rate limit, 액세스 토큰 수명·로그아웃 정책을 정의한다.
- 필요한 테스트: 요청 ID 응답·로그 전파, 잘못된 issuer 거부, 반복 로그인 제한 테스트.
- 포트폴리오 가치: 기능 구현 이후 운영 가능성을 고려한 경험을 보여준다.
- 예상 작업량: M

## 3. 잘 구현된 부분

1. 게시글·신고 목록은 `EntityGraph`로 N+1을 제거했고, `PostQueryPerformanceIntegrationTest`에서 게시글 수가 늘어도 SQL이 3회를 넘지 않도록 회귀 기준을 유지한다.
2. `V1__create_initial_schema.sql`에서 공감과 신고의 유니크 제약을 데이터베이스에도 두어 애플리케이션 검증만 의존하지 않았다.
3. 게시글 수정·삭제 권한은 서버의 작성자 ID로 검증해 프론트엔드 버튼 노출 여부에 의존하지 않는다.
4. 관리자 게시글 숨김은 관련 신고 해결, 처리자 기록, moderation 이력 저장을 하나의 트랜잭션으로 묶는다.
5. `application.yml`의 `ddl-auto=validate`와 Flyway `V1`~`V7`로 스키마 변경을 재현 가능한 버전으로 관리한다.

## 5. 현재 구현하지 않아도 되는 것

1. **마이크로서비스 분리**: 현재 트래픽과 팀 규모에서는 분산 트랜잭션과 배포 복잡도가 이점보다 크다.
2. **Kafka 또는 메시지 브로커**: 현재 작업은 단일 트랜잭션으로 충분하며 비동기 처리 요구가 없다.
3. **Kubernetes**: 무료·저비용 단일 서버 포트폴리오 운영에는 관리 비용이 과도하다.
4. **Elasticsearch/OpenSearch**: 데이터 규모와 검색 품질 요구가 확인되기 전에는 MySQL 검색 개선이 우선이다.
5. **Redis 캐시**: 단일 서버에서는 크기·TTL이 제한된 로컬 캐시로 충분하다. 다중 서버 전환 시 검토한다.

## 7. 검증 기록

- 최초 `.\gradlew.bat test --rerun-tasks`: 최초 리뷰 시점 테스트를 캐시 없이 재실행했다.
- 숨김 콘텐츠 재현: 상세 `404`, 후속 기록 `200` 및 1건 노출 확인 후 테스트 게시글 소프트 삭제.
- 제한 관리자 재현: 제한 처리 `200`, 제한된 기존 토큰의 metrics `200`, 자기 재활성화 `200` 확인 후 관리자 계정 재활성화.
- 내 기록 페이지 크기 재현: `GET /api/v1/posts/me?size=101`이 `200`으로 처리됨.
- 프론트엔드 마크다운: `react-markdown`을 사용하며 `rehypeRaw`와 `dangerouslySetInnerHTML`은 사용하지 않아 저장된 raw HTML이 바로 실행되는 근거는 발견하지 못했다.
- P1 수정 후 `.\gradlew.bat test --rerun-tasks`: 전체 14개 테스트 성공.
- 실제 MySQL P1 재검증: 숨김 후속 기록 `404`, 제한 관리자 조회·변경 `403`, 동시 공감·신고 각각 5건에서 카운터와 실제 행이 모두 5로 일치.
- 2026-07-24 현재: 백엔드 40개 테스트가 통과하며 MySQL 8.4에서 Flyway, 동시 공감·신고·Refresh 갱신, 상태 정책을 자동 검증한다.
- 설정 검증: 짧은 JWT 시크릿과 0 이하 요청 제한 용량은 애플리케이션 시작 단계에서 거부한다.

이 리뷰와 후속 테스트가 모든 운영 위험의 부재를 보장하지는 않는다. 장시간 부하, 실제 프록시·HTTPS 구성과 다중 인스턴스 환경은 별도 배포 환경에서 검증해야 한다.
