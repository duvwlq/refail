# Fail 프로젝트 기획서

## 1. 서비스 요약

### 한 줄 정의
실패를 공유함으로써 실패가 끝이 아니라 과정처럼 느껴지게 만드는 사이트

### 핵심 목적
- 실패에 대한 두려움을 줄인다.
- 큰 실패와 작은 실패 모두 편하게 말할 수 있게 만든다.
- 실패 이야기가 나중에 변화, 재도전, 성공으로 이어질 수 있게 한다.

### 감정적 방향
- 판단하는 공간이 아니다.
- 훈계하는 자기계발 공간이 아니다.
- 솔직함, 공감, 가벼운 위로, 다시 해보는 마음이 남는 공간이다.

## 2. 문제 정의

사람들은 보통 실패를 숨긴다. 이유는 다음과 같다.
- 실패는 창피하다고 느낀다.
- 작은 실패는 말하기 애매하다고 느낀다.
- 큰 실패는 너무 무겁고 부담스럽다고 느낀다.
- 성공을 보여주는 공간은 많지만 실패를 안전하게 말할 공간은 적다.

이 때문에 다음과 같은 문제가 생긴다.
- 사소한 좌절도 혼자 버티게 된다.
- 다른 사람들은 다 잘하고 있다고 착각하게 된다.
- 실패를 과정이 아니라 자기 자신에 대한 증명처럼 받아들이게 된다.

### 시대적 배경
SNS와 다양한 온라인 플랫폼이 발전하면서 사람들은 이전보다 훨씬 자주 타인의 일상을 보게 되었다.
그 과정에서 특히 많이 보이는 것은 성공, 행복, 성취, 잘 풀린 결과들이다.

하지만 실제 삶은 늘 성공적이지 않고, 모든 일상이 행복한 것도 아니다.
사람은 자주 실패하고, 흔들리고, 작은 걸림돌 앞에서 멈추기도 한다.

문제는 많은 사람들이 타인의 성공과 행복을 반복적으로 보면서 자신의 불안과 결핍을 더 크게 느끼게 된다는 점이다.
결국 실패는 더 숨기게 되고, 비교는 더 심해지며, 자신만 뒤처진 것 같은 감각이 커진다.

이 서비스는 바로 그 지점을 문제로 본다.
성공만 보이는 흐름 대신, 실패를 돌아보고 다시 나아가는 흐름을 보여주고자 한다.

## 3. 비전

### 이 서비스가 바꾸고 싶은 것
사용자가 이 서비스를 통해 다음과 같이 느끼게 하는 것이 목표다.
- 나만 이런 게 아니구나
- 이게 끝은 아니구나
- 이것도 나중에는 하나의 이야기가 될 수 있구나

### 장기 비전
실패를 하나의 낙인이 아니라 시간의 흐름으로 바꾼다.
- 실패
- 돌아봄
- 재도전
- 변화
- 성공 또는 진행 중

### 서비스가 보여주고 싶은 것
이 서비스의 목표는 불행을 전시하는 것이 아니다.

보여주고 싶은 것은 다음과 같다.
- 타인의 실패
- 그 실패를 돌아보는 태도
- 실패 이후 성장하는 과정
- 작은 걸림돌을 해결해 나가는 모습
- 결국 다시 시도하고 극복해 가는 흐름

즉, 성공한 결과만을 소비하게 하는 서비스가 아니라, 실패를 통해 배우고 다시 나아가는 과정을 보게 하는 서비스다.

### 서비스 철학
사람은 성공을 보면서도 배울 수 있지만, 실패를 통해 더 많이 배울 수도 있다.
실패는 단순히 부정적인 결과가 아니라, 같은 실수를 반복하지 않게 만드는 기록이 될 수 있다.

그래서 이 서비스는 자신의 실패를 적고 돌아보게 하며, 다른 사람의 실패와 변화 과정을 통해 각자의 해결점을 찾도록 돕는다.

중요한 것은 실패에 오래 머무르게 만드는 것이 아니라, 실패를 지나 극복과 변화로 이어지게 만드는 것이다.

## 4. 타깃 사용자

### 주요 사용자
- 성과 압박을 자주 느끼는 10대 후반부터 30대 사용자
- 학생, 취준생, 직장인, 창작자, 프리랜서, 창업 준비자
- 너무 무겁거나 형식적인 공간 말고 편하게 솔직해지고 싶은 사람

### 사용자 유형
1. 조용히 힘들어하는 사람
판단받지 않고 실패를 털어놓을 공간이 필요하다.

2. 일상 기록형 사용자
다이어트, 공부, 수면, 소비, 습관 같은 작은 실패를 기록하고 싶다.

3. 다시 돌아오는 사용자
나중에 다시 와서 그 이후에 무엇이 달라졌는지 말하고 싶다.

4. 공감형 독자
직접 자주 쓰진 않아도 다른 사람의 경험을 읽고 위로를 얻는다.

## 5. 핵심 가치 제안

### 이 서비스가 다른 이유
- 실패를 조롱거리로 소비하지 않는다.
- 심각한 실패와 사소한 실패를 같은 무게로 다룬다.
- 실패 글이 이후 변화와 성공까지 이어질 수 있다.
- 실패를 정체성이 아니라 과정으로 바라보게 만든다.

### 사용자에게 주는 가치
- 감정적 해소
- 비슷한 경험에서 오는 공감
- 시간에 따라 쌓이는 성장 기록
- 다시 시도할 용기

## 6. 브랜드 메시지

### 핵심 키워드
- 실패
- 솔직함
- 공감
- 재도전
- 과정
- 변화

### 톤앤매너
- 가볍고 편안하다
- 너무 무겁지 않지만 공감은 충분하다
- 솔직하지만 날카롭지 않다
- 일상적인 실패도 편하게 올릴 수 있다
- 응원하지만 훈계처럼 들리지 않는다

### 카피 후보
- 실패도 이야기의 일부입니다
- 실패를 말해도 괜찮은 곳
- 실패 다음의 다시 한 번
- 작은 실패도, 큰 실패도, 여전히 나눌 가치가 있다
- 실패는 이야기의 끝이 아니다

### 서비스 소개문 초안
우리는 너무 자주 타인의 성공과 행복을 본다.
하지만 실제 삶은 늘 잘 풀리지 않고, 누구나 실패하고 흔들린다.

이 서비스는 성공만 보이는 흐름 대신, 실패를 돌아보고 다시 나아가는 과정을 보여주기 위해 만들어진다.
큰 실패도, 일상의 작은 실패도 편하게 기록하고, 그 경험이 나중에 어떤 변화와 성장으로 이어졌는지 남길 수 있는 공간이다.

목표는 불행을 보여주는 것이 아니다.
실패를 통해 배우고, 같은 실수를 반복하지 않으며, 결국 다시 극복해 나가는 흐름을 함께 만드는 것이다.

## 7. 핵심 사용자 경험

### 경험 목표
사용자가 다음과 같이 느껴야 한다.
1. 글을 쓰기 쉽다
2. 글을 써도 안전하다
3. 누군가 이해해 준다
4. 이야기를 나중에 이어갈 수 있다

### 주요 흐름
1. 사용자가 사이트에 들어온다.
2. 공감 중심의 실패 글들을 본다.
3. 자신의 실패를 작성한다.
4. 다른 사용자들이 공감하거나 비슷한 경험을 나눈다.
5. 원 작성자가 나중에 다시 와서 후속 기록을 남긴다.
6. 실패가 하나의 여정으로 보이기 시작한다.

## 8. 콘텐츠 구조

### 주요 콘텐츠 유형
1. 실패 글
무엇이 잘되지 않았는지에 대한 원래 이야기

2. 업데이트 글
실패 이후 어떤 변화가 있었는지에 대한 후속 기록

3. 성공 후속 기록
독립된 성공 글이 아니라 기존 실패 글에 연결되는 결과 기록

### 실패 글 항목
- 제목
- 실패 내용
- 카테고리
- 실패 크기: 작음 / 보통 / 큼
- 감정 태그: 창피함 / 답답함 / 슬픔 / 무기력 / 다시 해보는 중
- 작성 방식: 익명 또는 닉네임
- 다시 해볼 의향 여부

### 업데이트 글 항목
- 원본 실패 글 연결
- 무엇이 달라졌는지
- 현재 상태: 아직 실패 중 / 다시 시도 중 / 나아지는 중 / 성공함
- 짧은 회고

## 10. 커뮤니티 원칙

### 반드시 지켜야 할 원칙
- 조롱과 망신주기를 금지한다.
- 실패의 크기로 사람을 평가하지 않는다.
- 모든 실패를 억지로 성공 서사로 몰아가지 않는다.
- 익명 작성과 닉네임 작성 모두 존중한다.

### 댓글 가이드
좋은 댓글 예시
- 나도 비슷한 경험이 있었어요
- 그 상황 많이 힘들었겠네요
- 공유해줘서 고마워요
- 저도 아직 그 단계에 있어요

좋지 않은 댓글 유형
- 비난
- 비꼼
- 원치 않는 강한 조언
- 남의 실패 앞에서 자기 성공만 드러내는 태도

### 댓글 기능에 대한 현재 판단
댓글은 장기적으로는 고려할 수 있지만, 현재 MVP에서는 보류한다.

이유는 다음과 같다.
- 사람들은 항상 선의로 반응하지 않는다
- 실패를 공유하는 공간에서 댓글은 쉽게 상처가 될 수 있다
- 초기에 댓글까지 열면 운영 부담이 급격히 커진다
- 서비스의 첫 목표는 토론보다 기록과 공감에 있다

## 10-1. 추천 닉네임 정책

### 추천 방향
혼합 정체성 모델을 사용한다.
- 글은 익명으로 올릴 수 있다
- 또는 닉네임 계정으로 올릴 수 있다

실명은 요구하지 않는다.

### 이 방식이 적합한 이유
- 익명은 무겁고 창피한 실패를 올릴 장벽을 낮춘다.
- 닉네임은 사용자가 다시 돌아와 업데이트를 남길 수 있게 한다.
- 실명은 서비스 목적상 부담을 키우고 위축을 만든다.

### MVP 기준 계정 구조
- 읽기는 로그인 없이 가능
- 글쓰기와 댓글은 회원가입 필요
- 가입한 사용자는 하나의 기본 닉네임을 가진다
- 글마다 익명 또는 닉네임 공개를 선택할 수 있다

### 추천 닉네임 규칙
- 계정당 기본 닉네임은 하나
- 닉네임 변경은 예를 들어 30일에 1회로 제한
- 가능하면 닉네임 중복을 막고, 어렵다면 추후 구분 장치를 둔다
- 욕설성, 사칭성 닉네임은 금지
- 인기 경쟁을 만드는 과한 프로필 꾸미기는 제외

### 공개 방식 규칙
- 익명 글은 다른 사용자에게 닉네임이 보이지 않는다
- 작성자는 자신의 계정 안에서 익명 글도 관리할 수 있다
- 사용자는 개인 화면에서 익명 글과 닉네임 글을 모두 볼 수 있다
- 공개 프로필이 생기더라도 닉네임 글만 노출한다

### MVP에서 피할 것
- 한 계정에서 여러 페르소나 운영
- 팔로워 수
- 공개 평판 점수
- 실명 인증
- 익명 글의 공개 프로필 노출

## 10-2. 추천 댓글 및 공감 규칙

### 추천 방향
반응 구조를 두 층으로 나눈다.
- 아주 쉽게 누를 수 있는 공감 리액션
- 더 신중하게 남기는 댓글

단, 현재 MVP에서는 댓글 기능을 열지 않고 공감 리액션 중심으로 운영한다.

### 공감 리액션 설계
일반적인 좋아요 대신 지지형 반응을 사용한다.

추천 리액션
- 나도 그래
- 응원 보낼게요
- 공유해줘서 고마워요
- 다음 시도를 응원해요

이유
- 좋아요는 실패 콘텐츠에서 어색하거나 가볍게 느껴질 수 있다
- 리액션 문구 자체가 커뮤니티 문화를 만든다

### 댓글 정책 추천
댓글은 추후 확장 기능으로 검토하고, MVP에서는 제외한다.

### 댓글 규칙
- 사람을 평가하지 말고 경험에 반응한다
- 공감을 먼저, 조언은 나중에 둔다
- 조언은 원할 때만 한다
- 비꼼, 비교, 훈계형 말투를 금지한다
- 작은 실패를 하찮다고 공격하지 않는다
- 큰 실패를 어리석다고 공격하지 않는다

### 제품 내 기본 규칙
댓글 기능이 도입되면 첫 댓글 작성 전 짧은 안내를 보여준다.
- 다정하게 말하기
- 판단하지 않기
- 조언보다 경험 공유 먼저 하기

### MVP 모더레이션 장치
- 글 신고
- 금칙어 및 명백한 비방어 차단
- 신고된 콘텐츠 검토
- 관리자 삭제 및 계정 제한 기능

### 조언 처리 방식 추천
글 작성 시 선택 항목을 둔다.
- 위로가 필요해요
- 조언도 괜찮아요

이 항목은 추후 댓글 기능이 열릴 때 반응 톤을 안내하는 기준으로도 활용할 수 있다.

### 정렬 기준 추천
- 댓글 기능 도입 전까지는 공감 수를 주요 반응 지표로 사용한다
- 댓글 기능이 생기더라도 과격함이나 논쟁성이 아니라 공감 품질 중심으로 정렬한다
- 작성자 업데이트가 있으면 일반 댓글보다 위에 보이게 한다

### MVP에서 피할 것
- 싫어요 버튼
- 웃김 반응
- 인용 재게시
- 단순 반응 수 기반 댓글 랭킹
- 로그인 없는 익명 댓글
- 댓글 기능의 조기 도입

### 기본 안전 원칙
- 글은 익명으로 올릴 수 있다
- 댓글은 현재 보류한다
- 반복적인 악용은 빠르게 제한한다
- 초기에는 활동량보다 신뢰가 더 중요하다

## 15. 저비용 운영 전략

### 운영 원칙
초기 버전은 1명이 감당할 수 있을 만큼 작아야 한다.

### 비용을 줄이는 제품 결정
- MVP 기능 범위를 좁게 유지한다
- 이미지나 영상보다 텍스트 중심으로 간다
- 읽기는 자유롭게, 작성은 로그인 후에만 허용한다
- 운영 부담이 큰 소셜 기능은 줄인다
- 성공 글은 독립 게시글이 아니라 후속 기록으로만 둔다

### 비용을 줄이는 모더레이션 결정
- 처음부터 명확한 커뮤니티 규칙을 둔다
- 글 신고 기능을 둔다
- 명백한 금칙어는 자동 차단한다
- 악용 반복 사용자는 복잡한 페널티보다 빠른 제한을 적용한다
- 익명 글만 허용하고 댓글은 MVP에서 제외한다

### 비용을 줄이는 기술 결정
- 고정비가 낮은 배포 방식을 선택한다
- 관리형 DB를 사용한다
- 실시간 채팅, 영상, 대용량 업로드는 MVP에 넣지 않는다
- 텍스트 중심 데이터 구조를 유지한다
- 지원 부담이 적은 로그인 방식만 사용한다

### 콘텐츠 정책으로 줄일 수 있는 비용
- MVP에는 이미지 첨부를 제외하거나 최소화한다
- DM 기능을 넣지 않는다
- 팔로우 그래프를 만들지 않는다
- 과도한 프로필 꾸미기를 넣지 않는다
- 복잡한 추천 피드를 만들지 않는다

### 추천 운영 도구
- 신고 관리용 관리자 화면
- 글 숨김 / 복구
- 계정 제한 / 정지
- 간단한 카테고리 관리

### 추천 출시 방식
작은 커뮤니티 크기에서 시작한다.
- 소규모 공개
- 초기 사용자 중심 운영
- 글 분위기 관찰
- 규칙 조정 후 확장

### 비용을 키우는 대표 요소
- 이미지 / 영상 업로드
- 모든 기능의 완전 익명화
- 실시간 기능
- 복잡한 소셜 구조
- 약한 모더레이션

## 16. 저비용 기술 방향

### 기본 기술 방향
스프링 부트 백엔드를 사용하는 전제에서 가장 현실적인 구조는 다음과 같다.
- 정적 또는 하이브리드 프론트엔드
- 작은 규모의 스프링 부트 API 서버 1개
- 단순한 인증 구조
- 관리형 데이터베이스

### 필요한 구조적 특성
- 낮은 트래픽에서도 효율적일 것
- 상시 큰 고정비가 들지 않을 것
- 관리자 도구를 만들기 쉬울 것
- 배포가 단순할 것
- 혼자 유지보수 가능할 것

### 실전 스택 예시
옵션 A
- Next.js 프론트엔드
- Spring Boot 백엔드
- MySQL 8
- 프론트는 Vercel
- 백엔드는 Render 또는 Railway

옵션 B
- React 프론트엔드
- Spring Boot 백엔드
- MySQL 8
- 프론트는 Cloudflare Pages
- 백엔드는 소형 VPS 1대

옵션 C
- Next.js 프론트엔드
- Spring Boot 백엔드
- Supabase Postgres만 사용
- 프론트는 Vercel
- 백엔드는 Render

### 추천 조합
Next.js + Spring Boot + MySQL

추천 이유
- 프론트와 백엔드 역할이 분명하다
- 스프링 부트는 이후 기능 확장에 안정적이다
- 텍스트 중심 커뮤니티에는 복잡한 인프라가 필요 없다
- 프론트는 저렴하게 운영 가능하다
- 백엔드는 초기에 작은 인스턴스 하나로 충분하다
- 관리자 기능과 모더레이션 기능 추가가 쉽다

### DB 비용 전략
- 스키마를 단순하게 유지한다
- 모든 행동을 이벤트 로그로 과하게 남기지 않는다
- 필요한 카운터만 비정규화한다
- 처음부터 페이지네이션을 넣는다
- 삭제는 완전 삭제보다 소프트 삭제를 우선한다

### 스프링 부트 비용 전략
- MVP에서는 단일 애플리케이션으로 간다
- 인증, 관리자, 피드, 모더레이션을 서비스 단위로 쪼개지 않는다
- 불필요한 리액티브 복잡성은 넣지 않는다
- 스프링 시큐리티는 로그인과 역할 관리 중심으로만 사용한다
- 명확히 필요할 때만 스케줄 작업을 추가한다
- Redis, 메시지 큐, 검색 엔진은 넣지 않는다

### 추천 배포 구조
- 프론트는 저비용 정적 호스팅
- 백엔드는 스프링 부트 단일 인스턴스
- DB는 관리형 MySQL
- 첨부파일이 필요해질 때만 오브젝트 스토리지 추가

### 추천 인증 방향
- 이메일 회원가입 또는 단순 소셜 로그인
- 역할은 user, admin 정도로 단순화
- MVP에서는 복잡한 권한 체계 제외

### MVP에서 넘지 말아야 할 인프라 경계
- 마이크로서비스 금지
- Kubernetes 금지
- 별도 검색 엔진 금지
- 추천 엔진 금지
- 필요 없는 백그라운드 작업 금지
- 별도 인증 서버 금지

다음처럼 보여줘야 한다.

- 정책과 기능 설계가 강하게 연결된 제품

### 강한 포트폴리오 포인트
- 실패에 대한 두려움이라는 실제 감정 문제를 출발점으로 삼았다
- 감정 목표를 제품 정책으로 번역했다
- 익명성, 연속성, 안전성의 균형을 설계했다
- 혼자 만들 수 있는 범위로 현실적으로 스코프를 잡았다
- 제품 복잡도에 맞는 백엔드 구조를 선택했다
