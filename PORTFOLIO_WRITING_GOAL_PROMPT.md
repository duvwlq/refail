## 실행 프롬프트

당신은 백엔드 개발자 채용을 위한 기술 포트폴리오를 작성하는 시니어 백엔드 엔지니어이자 기술 에디터다.

작업 저장소는 `D:\Codex_Project\Fail`이며 프로젝트 이름은 `Re:Fail`이다.

### 최종 목표

저장소의 코드, 테스트, Git 이력과 한글 문서를 근거로 백엔드 개발자 포트폴리오 초안을 작성한다. 기능 목록을 나열하지 말고 다음 역량이 드러나는 문제 해결 중심 문서로 완성한다.

- 사용자 문제를 기술 요구사항으로 전환한 능력
- 데이터 정합성과 동시성 문제를 재현하고 해결한 능력
- JPA와 MySQL 조회 성능을 측정하고 회귀를 방지한 능력
- JWT 인증 수명주기와 권한 회수를 설계한 능력
- 운영 네트워크, 관측성, 테스트와 CI를 연결한 능력
- 현재 규모에 맞춰 기술을 선택하고 과도한 설계를 피한 판단력

최종 산출물은 모두 한글로 작성한다.

### 반드시 읽을 근거

작업 전에 다음 문서를 읽고 서로 모순되는 수치나 완료 상태가 없는지 확인한다.

1. `README.md`
2. `PORTFOLIO_PROBLEM_SOLVING.md`
3. `PORTFOLIO_IMPROVEMENTS.md`
4. `PORTFOLIO_POINTS.md`
5. `PERFORMANCE.md`
6. `BACKEND_REVIEW_RESULT.md`
7. `AUTH_SECURITY.md`
8. `ARCHITECTURE.md`
9. `DEPLOYMENT.md`
10. `REFACTORING_LOG.md`
11. `DEMO_SCENARIO.md`
12. `DEMO_VIDEO_SCRIPT.md`

수치나 동작을 인용할 때는 관련 코드, 테스트, 스크립트와 Git 이력도 확인한다. 문서만 보고 사실을 추정하지 않는다.

### 사실성 규칙

1. 저장소에서 확인되지 않은 기능, 사용자 수, 트래픽, 운영 장애와 협업 경험을 만들지 않는다.
2. 로컬 Docker Desktop과 MySQL 8.4에서 측정한 결과를 실제 운영 성과로 표현하지 않는다.
3. 공개 도메인을 사용한 외부 배포, 무중단 배포, 장시간 부하 테스트와 다중 서버 운영을 완료했다고 쓰지 않는다.
4. `대용량`, `완벽한 보안`, `완전한 해결`, `고가용성`처럼 증명 범위를 넘는 표현을 사용하지 않는다.
5. 성능 수치에는 데이터 규모, 측정 환경과 반복 조건을 함께 적는다.
6. 백분율은 원본 수치로 다시 계산하고 계산식을 작업 기록에 남긴다.
7. 테스트 개수와 CI 상태는 현재 저장소에서 다시 확인한 값만 사용한다.
8. 한계와 다음 검증 항목을 숨기지 않는다.

### 핵심 사실 기준선

다음 값은 문서에서 확인해야 하는 기준선이며, 현재 코드나 최신 검증 결과가 다르면 최신 근거로 수정한다.

- 서비스 문구: `Re:Fail - 실패를 공유하고, 다음을 준비하다`
- 백엔드: Java 21, Spring Boot 3.5.3, Spring Data JPA, Spring Security
- 데이터베이스: MySQL 8, Flyway
- 검색: 5만 건 본문 검색 평균 `72.47ms → 6.57ms`, p95 `105.49ms → 7.84ms`
- 관리자 지표: API SQL `7회 → 2회`, 집계 평균 `6.75ms → 3.18ms`
- 목록 조회: 게시글 수와 무관하게 SQL 최대 3회
- 동시성: MySQL 8.4에서 공감·신고 각각 8개 동시 요청의 카운터와 실제 행 개수 일치
- 인증: 15분 Access Token, 해시 저장·회전형 Refresh Token, 재사용 탐지와 계열 폐기
- 자동 검증 기준: 백엔드 40개, Playwright P0 3개, HTTPS 운영 스모크 10단계
- 운영 경계: Caddy만 HTTP·HTTPS 공개, DB·백엔드·프론트엔드 호스트 포트 비노출
- 남은 범위: 실제 외부 서버·공개 도메인 배포와 백업 복구 훈련

### 문제 해결 서술 형식

각 대표 사례는 반드시 아래 순서로 작성한다.

1. **문제**: 어떤 현상과 위험을 발견했는가
2. **원인**: 코드, SQL, 잠금, 인증 또는 네트워크 관점의 원인은 무엇인가
3. **제약**: 프로젝트 규모, 비용, 운영 범위와 유지보수 조건은 무엇인가
4. **대안**: 검토한 선택지와 선택하지 않은 이유는 무엇인가
5. **구현**: 실제로 어떤 구조와 코드를 변경했는가
6. **검증**: 어떤 테스트, 데이터, 반복 횟수와 실패 조건으로 확인했는가
7. **결과**: 수치나 동작이 어떻게 달라졌는가
8. **한계**: 아직 증명하지 못한 범위와 다음 검증은 무엇인가

### 반드시 포함할 대표 사례

다음 중 최소 5개를 본문에 포함하되, 성능·정합성·보안·운영 영역이 모두 한 번 이상 등장해야 한다.

1. JPA N+1 제거와 목록 SQL 최대 3회 회귀 테스트
2. MySQL ngram FULLTEXT로 5만 건 검색 성능 개선
3. 공감·신고 lost update와 InnoDB 교착 상태 해결
4. JWT Refresh Token 회전·재사용 탐지와 관리자 권한 즉시 회수
5. 사용자 검색어 기반 무제한 로컬 캐시 제거
6. 관리자 운영 지표 SQL 7회에서 2회로 축소
7. H2와 Testcontainers MySQL을 나눈 계층형 테스트 전략
8. Caddy HTTPS 단일 진입점과 10단계 운영 배포 스모크

### 산출물

다음 파일을 새로 작성하거나 최신 상태로 갱신한다.

#### 1. `PORTFOLIO_DRAFT.md`

채용 담당자와 백엔드 면접관이 5분 안에 핵심을 파악할 수 있는 본문이다.

구성:

1. 프로젝트 한 줄 소개
2. 문제 정의와 서비스 목표
3. 담당 범위와 기술 스택
4. 시스템 아키텍처와 기술 선택
5. 대표 문제 해결 사례 5개
6. 테스트·CI·운영 준비
7. 성과 요약 표
8. 한계와 다음 단계
9. GitHub 및 시연 자료

각 문제 해결 사례는 250~450자 내외로 작성하고, 가장 중요한 수치를 첫 두 문장 안에 배치한다.

#### 2. `PORTFOLIO_ONE_PAGE.md`

한 페이지 이력서나 Notion 상단에 사용할 압축본이다.

포함 내용:

- 80자 이내 프로젝트 소개
- 담당 역할
- 핵심 성과 4개
- 기술 스택
- 대표 문제 해결 3개
- GitHub 링크

#### 3. `PORTFOLIO_INTERVIEW.md`

면접 답변용 문서다.

아래 질문에 대해 각각 30초와 1분 답변을 작성한다.

- 왜 이 서비스를 만들었는가
- 왜 완전 익명이나 일반 댓글을 선택하지 않았는가
- 가장 어려웠던 백엔드 문제는 무엇이었는가
- 동시성 문제를 어떻게 재현하고 해결했는가
- FULLTEXT를 선택하고 Elasticsearch를 도입하지 않은 이유는 무엇인가
- JWT인데 왜 DB 상태를 다시 조회하는가
- 왜 모놀리식 Spring Boot를 선택했는가
- 테스트 전략을 어떻게 구성했는가
- 운영 배포에서 아직 부족한 점은 무엇인가

#### 4. `PORTFOLIO_EVIDENCE_MATRIX.md`

포트폴리오 주장과 근거를 연결한다.

| 주장 | 코드·설정 | 자동 테스트 | 측정·문서 | 표현 가능한 범위 |
| --- | --- | --- | --- | --- |

근거가 없는 주장은 본문에서 제거하거나 `향후 계획`으로 이동한다.

### 단계별 실행 계획

#### 1단계: 기준선 확인

- `git status`, 최근 커밋과 현재 브랜치를 확인한다.
- 기존 사용자 변경을 덮어쓰지 않는다.
- 문서의 기술 버전, 테스트 개수, 성능 수치와 완료 상태를 비교한다.
- 모순이 있으면 코드와 최신 자동 검증을 우선하고 수정 이유를 기록한다.

완료 조건:

- 사용할 수치와 주장 목록이 확정된다.
- 각 주장에 최소 하나의 저장소 근거가 연결된다.

#### 2단계: 문제 해결 사례 선별

- 백엔드 채용 관점에서 가치가 높은 사례를 우선순위화한다.
- 비슷한 사례는 하나로 통합하고 기능 소개와 문제 해결 사례를 구분한다.
- 각 사례의 문제, 원인, 선택, 검증과 한계를 메모한다.

완료 조건:

- 성능, 정합성, 보안, 운영을 포함한 대표 사례 5개가 선정된다.

#### 3단계: 포트폴리오 초안 작성

- `PORTFOLIO_DRAFT.md`와 `PORTFOLIO_ONE_PAGE.md`를 작성한다.
- 첫 화면에는 서비스 설명보다 백엔드 문제 해결 성과가 먼저 보이게 한다.
- 기술명 나열보다 판단과 검증 근거를 강조한다.
- 모든 문장은 존댓말 서술형 또는 간결한 명사형 중 하나로 통일한다.

완료 조건:

- 5분용 본문과 한 페이지 요약이 서로 같은 수치와 메시지를 사용한다.

#### 4단계: 면접 답변과 근거표 작성

- `PORTFOLIO_INTERVIEW.md`를 작성한다.
- `PORTFOLIO_EVIDENCE_MATRIX.md`에서 모든 핵심 주장에 코드·테스트·문서 근거를 연결한다.
- 답변은 결론부터 말하고 구현 세부사항과 한계를 뒤에 배치한다.

완료 조건:

- 면접 질문마다 30초와 1분 답변이 있다.
- 근거가 없는 완료형 표현이 없다.

#### 5단계: 사실 검증과 품질 점검

- 성능 개선율을 원본 수치로 다시 계산한다.
- 테스트를 실행해 현재 성공 개수를 확인한다.
- 프론트엔드 lint·build와 핵심 E2E·운영 스모크의 최신 CI 상태를 확인한다.
- Markdown 내부 링크와 표 형식을 검사한다.
- 실제 외부 배포를 완료한 것처럼 보이는 표현이 없는지 검색한다.

완료 조건:

- 모든 수치가 근거 문서와 일치한다.
- 과장 표현과 확인되지 않은 경험이 없다.
- 문서 링크가 유효하다.

#### 6단계: 문서 연결과 Git 마무리

- `README.md`와 `DOCUMENT_INDEX.md`에 새 포트폴리오 문서를 연결한다.
- `git diff --check`와 전체 변경을 검토한다.
- 문서 작업과 검증 보완을 논리적인 단위로 커밋한다.
- 원격에 푸시하고 GitHub Actions 성공을 확인한다.

완료 조건:

- 작업 트리가 깨끗하다.
- 로컬 HEAD와 원격 브랜치가 일치한다.
- 원격 CI가 성공한다.

### 작성 스타일

- 모든 문서는 한글로 작성한다.
- 첫 문장에서 결론과 결과를 먼저 제시한다.
- 문장은 짧게 쓰고 한 문단에는 하나의 주장만 둔다.
- `제가`, `열심히`, `많이 배웠다`보다 관찰·선택·검증 사실을 쓴다.
- 기술 용어는 필요한 범위에서만 영문을 병기한다.
- 표는 비교와 수치 요약에만 사용한다.
- 같은 성과를 여러 섹션에서 반복하지 않는다.
- 채용 담당자가 세부 코드를 몰라도 문제와 결과를 이해할 수 있게 쓴다.
- 면접관이 깊게 질문할 수 있도록 구체적인 구현 단서를 남긴다.

### 최종 보고 형식

작업 완료 후 다음만 간결하게 보고한다.

1. 생성·수정한 문서
2. 포트폴리오의 대표 문제 해결 사례
3. 실행한 검증과 결과
4. 사실성 검토에서 제외하거나 낮춘 주장
5. 커밋과 원격 CI 상태
6. 다음 작업 한 가지

중간 단계에서 실패한 테스트나 모순을 숨기지 말고 원인과 조치를 기록한다. 완료 조건을 충족하기 전에는 목표를 완료 처리하지 않는다.
