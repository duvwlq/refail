# Re:Fail 백엔드 부족한 점 점검 프롬프트

아래 프롬프트를 코드 리뷰가 가능한 AI 도구에 그대로 입력한다. 프로젝트 루트에서 실행하면 현재 코드와 테스트를 직접 확인하도록 설계했다.

---

## 전체 리뷰 프롬프트

```text
당신은 실무 경력 10년 이상의 시니어 Java/Spring 백엔드 개발자이자 기술 면접관이다.

현재 작업 공간의 Re:Fail 프로젝트를 실제로 탐색하고, 백엔드 개발자 포트폴리오 관점에서 부족한 점을 냉정하게 코드 리뷰하라. Re:Fail은 실패 경험을 익명 또는 닉네임으로 기록하고 후속 기록을 통해 성장 과정을 남기는 포트폴리오용 서비스다.

현재 기술 구성은 다음과 같다.
- Java 21, Spring Boot 3
- Spring Data JPA, Spring Security
- JWT 인증
- MySQL 8, Flyway
- Swagger/OpenAPI
- JUnit 5 통합 테스트와 H2 MySQL 호환 모드
- Docker Compose 기반 로컬 실행
- Spring Boot 단일 서버 구조

이미 구현된 주요 기능은 다음과 같다.
- 회원가입, 로그인, 내 정보 조회
- 게시글 작성·조회·수정·삭제
- 익명·닉네임 공개 선택
- 마크다운 게시글과 다음 시도 계획
- 후속 기록 CRUD와 성장 상태
- 공감 반응과 본인 글 공감 제한
- 신고, 관리자 신고 조회, 게시글 숨김·복구
- 카테고리 CRUD, 검색, 정렬, 필터, 페이지네이션
- Flyway 스키마 마이그레이션
- 게시글·신고 목록 N+1 개선
- 카테고리 로컬 캐시와 조회용 복합 인덱스

리뷰 원칙:
1. 문서의 주장만 믿지 말고 실제 소스 코드, 설정, 마이그레이션, 테스트를 확인하라.
2. 추측으로 문제를 만들지 말고 모든 지적에 파일 경로와 줄 번호 또는 구체적인 코드 근거를 제시하라.
3. 기능 개수가 아니라 정합성, 보안, 장애 대응, 운영 가능성, 테스트 신뢰도를 우선 평가하라.
4. 포트폴리오 규모에 불필요한 마이크로서비스, Kubernetes, Kafka 같은 기술을 근거 없이 추천하지 마라.
5. 이미 잘 구현된 부분은 중복해서 개선 과제로 제시하지 마라.
6. 실제 장애나 데이터 손상이 가능한 문제와 단순 개선 사항을 명확히 구분하라.
7. 문제가 없으면 억지로 만들지 말고 검증하지 못한 잔여 위험이라고 표시하라.

다음 영역을 반드시 점검하라.

1. 도메인과 데이터 정합성
- 공감 수, 신고 수 같은 비정규화 집계 값이 동시 요청에서도 정확한가?
- 트랜잭션 경계와 잠금 전략이 적절한가?
- 소프트 삭제, 숨김, 사용자 제한 상태가 모든 조회·변경 경로에서 일관되게 적용되는가?
- 데이터베이스 제약 조건과 애플리케이션 검증이 서로 보완되는가?
- 중복 요청과 멱등성 문제가 있는가?

2. 인증과 보안
- JWT 만료, 위조, 잘못된 서명, 비활성 사용자 처리가 안전한가?
- Refresh Token, 로그아웃, 토큰 탈취 대응이 필요한 수준인가?
- 관리자 권한과 게시글 소유권 검증을 우회할 수 있는 API가 있는가?
- CORS, 비밀번호 저장, 민감 정보 로그, 기본 시크릿 설정에 문제가 있는가?
- 입력값으로 XSS, SQL Injection, 과도한 요청, 마크다운 스크립트 실행이 가능한가?
- Swagger와 운영용 API가 배포 환경에서 과도하게 노출되는가?

3. API 설계
- 상태 코드와 에러 응답이 일관적인가?
- 요청 검증과 문자열 길이 제한이 데이터베이스 스키마와 일치하는가?
- 페이지네이션 최대 크기, 정렬 파라미터, 검색어 길이가 제한되는가?
- 수정 API의 PUT/PATCH 의미와 응답 구조가 적절한가?
- 삭제, 신고, 공감 API가 재요청에 안전한가?
- 엔티티가 API 응답으로 직접 노출되는 부분이 있는가?

4. JPA와 MySQL
- 남아 있는 N+1 쿼리나 불필요한 전체 조회가 있는가?
- fetch join 또는 EntityGraph와 페이지네이션 사용이 안전한가?
- 인덱스가 실제 WHERE·ORDER BY 조건과 맞는가?
- `%keyword%` 검색과 운영 지표 집계가 데이터 증가 시 병목이 되는가?
- 읽기 전용 트랜잭션, batch 설정, 영속성 컨텍스트 사용이 적절한가?
- H2 테스트가 MySQL 전용 동작 차이를 놓칠 가능성이 있는가?

5. 동시성과 장애 대응
- 동시에 공감, 신고, 수정, 숨김 요청이 들어오면 lost update가 발생하는가?
- 유니크 제약 위반이 사용자 친화적인 API 오류로 변환되는가?
- 외부 또는 데이터베이스 장애 시 타임아웃과 예외 처리가 적절한가?
- 재시도 때문에 데이터가 중복 생성될 가능성이 있는가?
- 낙관적 잠금, 비관적 잠금, 원자적 UPDATE 중 어떤 방식이 필요한가?

6. 테스트 품질
- 성공 경로뿐 아니라 권한 실패, 중복 요청, 경계값, 동시성도 검증하는가?
- 테스트가 구현 세부사항에 지나치게 결합되어 있지 않은가?
- 쿼리 수 테스트가 안정적이고 실제 회귀를 잡을 수 있는가?
- MySQL Testcontainers 테스트가 필요한 핵심 영역은 무엇인가?
- 현재 테스트가 통과해도 운영에서 실패할 수 있는 시나리오는 무엇인가?

7. 운영 가능성
- 구조화 로그, 요청 추적 ID, 메트릭, 장애 확인 수단이 있는가?
- 헬스 체크가 단순 프로세스 상태와 데이터베이스 준비 상태를 구분하는가?
- 환경 변수와 프로필별 설정이 안전한가?
- Flyway 실패와 배포 롤백을 고려했는가?
- 관리자 작업 감사 로그가 충분한가?
- 무료 또는 저비용 단일 서버 운영에 맞는 개선인가?

8. 코드 품질과 구조
- 컨트롤러, 서비스, 리포지토리 책임이 적절히 분리되어 있는가?
- 도메인별 패키지 구조가 유지되고 순환 의존성이 없는가?
- 중복된 사용자·게시글 조회와 권한 검증을 안전하게 정리할 수 있는가?
- enum, DTO, 예외 코드 이름과 의미가 일관적인가?
- 과도하게 큰 서비스 또는 변경 이유가 서로 다른 코드가 결합되어 있는가?

출력 형식:

## 1. 총평
- 현재 백엔드 완성도를 신입 백엔드 포트폴리오 기준 100점 만점으로 평가하라.
- 점수 근거를 5문장 이내로 설명하라.
- 현재 가장 큰 위험 3개를 요약하라.

## 2. 발견 사항
심각도 순서로 작성한다.

각 항목은 반드시 다음 형식을 사용한다.

### [P0/P1/P2/P3] 문제 제목
- 근거: 파일 경로와 줄 번호, 관련 코드 설명
- 문제 상황: 어떤 요청 또는 데이터에서 발생하는가
- 영향: 보안, 데이터 정합성, 성능, 운영 중 무엇에 영향을 주는가
- 재현 방법: 테스트나 API 호출로 확인하는 방법
- 권장 수정: 현재 구조에서 가장 단순하고 안전한 해결책
- 필요한 테스트: 수정 후 추가할 테스트
- 포트폴리오 가치: 이 개선을 설명했을 때 보여줄 수 있는 역량
- 예상 작업량: S(2시간 이내), M(반나절~1일), L(2일 이상)

심각도 기준:
- P0: 즉시 데이터 손상, 인증 우회 또는 서비스 전체 장애 가능
- P1: 실제 사용자 환경에서 높은 확률로 발생할 정합성·보안 문제
- P2: 데이터 증가나 운영 과정에서 문제가 될 설계·성능·테스트 부족
- P3: 가독성, 유지보수성, 문서화 같은 품질 개선

## 3. 잘 구현된 부분
- 실제 코드 근거가 있는 강점만 최대 5개 작성하라.
- 일반적인 칭찬은 제외하라.

## 4. 우선순위 로드맵
- 지금 바로: 최대 3개
- 포트폴리오 제출 전: 최대 5개
- 실제 운영 전: 최대 5개
- 각 항목에 선택 이유와 선행 조건을 적어라.

## 5. 구현하지 않아도 되는 것
- 현재 프로젝트 규모에서 과도한 기술이나 기능을 최대 5개 제시하라.
- 왜 지금 필요하지 않은지 설명하라.

## 6. 다음 작업 1개
- 가장 먼저 구현할 작업 하나만 선택하라.
- 선택 이유, 수정 파일, 테스트 범위, 완료 조건을 구체적으로 제시하라.

리뷰를 시작하기 전에 프로젝트 파일 구조, 빌드 설정, application 설정, 엔티티, 리포지토리, 서비스, 보안 코드, Flyway SQL, 테스트를 직접 확인하라. 코드를 읽지 않고 일반론으로 답하지 마라.
```

## 특정 영역 재검토 프롬프트

전체 리뷰 후 한 영역을 더 깊게 검토할 때 사용한다.

```text
앞선 Re:Fail 백엔드 리뷰 중 `[검토할 영역 또는 발견 사항]`만 다시 집중적으로 검토하라.

반드시 실제 코드를 다시 확인하고 다음 내용을 제시하라.
1. 현재 요청 흐름과 트랜잭션 경계를 단계별로 설명
2. 정상·실패·동시 요청 시나리오
3. 실제 발생 가능한 문제와 발생 조건
4. 선택 가능한 해결책 최소 2개와 장단점
5. 현재 포트폴리오 규모에 가장 적합한 해결책 하나
6. 수정할 파일과 메서드
7. 먼저 작성할 실패 테스트
8. 구현 완료 조건

일반론이나 새로운 인프라 추천보다 현재 Spring Boot 단일 서버와 MySQL 구조에서 해결하는 방법을 우선하라.
```

## 리뷰 결과를 작업으로 바꾸는 프롬프트

```text
앞선 Re:Fail 백엔드 리뷰 결과를 기반으로 가장 우선순위가 높은 미해결 문제 하나를 선택하라.

다음 순서로 진행하라.
1. 해당 문제가 실제 코드에 존재하는지 다시 확인
2. 실패를 재현하는 테스트 먼저 작성
3. 현재 구조를 과도하게 변경하지 않는 최소 수정 구현
4. 관련 테스트와 전체 테스트 실행
5. 변경 전 문제, 선택한 해결책, 검증 결과를 `PORTFOLIO_IMPROVEMENTS.md`에 추가

기존 사용자 변경을 되돌리지 말고, 관련 없는 리팩터링을 섞지 마라. 작업을 계획만 제시하고 멈추지 말고 구현과 검증까지 완료하라.
```

## 사용 권장 순서

1. `전체 리뷰 프롬프트`로 코드베이스 전체를 점검한다.
2. P0와 P1 발견 사항이 실제 문제인지 `특정 영역 재검토 프롬프트`로 확인한다.
3. 검증된 문제는 `리뷰 결과를 작업으로 바꾸는 프롬프트`로 하나씩 구현한다.
4. 완료한 개선은 `PORTFOLIO_IMPROVEMENTS.md`에 누적한다.
5. 모든 개선을 한 번에 구현하지 말고 데이터 정합성, 보안, 테스트, 운영 순서로 진행한다.
